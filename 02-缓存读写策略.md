# 缓存读写策略 —— 深度详解

---

## 前言：为什么需要缓存读写策略？

缓存与数据库分属两个独立系统，天然存在**数据一致性**问题。三种经典的读写策略，本质上是在 **性能** 和 **一致性** 之间做不同的取舍。

---

## 1. Cache-Aside（旁路缓存）—— 最常用的策略

### 1.1 读流程

```
┌──────────┐   ①请求数据    ┌──────────┐
│          │──────────────▶│          │
│  Client  │               │  Cache   │
│          │◀─────── ✗ ────│ (Miss)   │
└──────────┘               └──────────┘
                                │
                          ②查数据库
                                ▼
                           ┌──────────┐
                           │    DB    │
                           └────┬─────┘
                                │ ③拿到数据
                                ▼
                           ┌──────────┐
                     ④写入 │  Cache   │
                           └────┬─────┘
                                │ ⑤返回数据
                                ▼
                           ┌──────────┐
                           │  Client  │
                           └──────────┘
```

**伪代码：**
```
func read(key):
    data = cache.get(key)
    if data != nil:
        return data              // 命中缓存，直接返回
    
    data = db.query(key)         // 缓存未命中，查数据库
    cache.set(key, data)         // 回填缓存
    return data
```

**关键点：**
- 读操作由**应用程序**来控制缓存回填，缓存层本身不主动加载数据。
- 缓存没命中时才查库，这叫 **Lazy Loading（懒加载）**。

### 1.2 写流程

```
func write(key, value):
    db.update(key, value)        // ① 先更新数据库
    cache.delete(key)            // ② 再删除缓存（不是更新！）
```

**为什么是删除缓存而不是更新缓存？**

| 做法 | 问题 |
|------|------|
| 更新缓存 | ① 如果是复杂计算结果，更新成本高；② 写入后可能再没人读，白更新；③ **并发写**时缓存更新顺序可能乱掉 |
| 删除缓存 | 删除是幂等的，成本极低；后续读请求触发 lazy load，自然回填最新值 |

**面试必问：为什么先更新数据库后删除缓存？反过来会怎样？**

```
方案 A：先删缓存，再更新数据库（错误方案）

Thread-A: 删除缓存 ✓
Thread-B: 读请求 → 缓存 miss → 查库(旧值) → 回填缓存(旧值) ✓
Thread-A: 更新数据库(新值) ✓

结果：数据库是新值，缓存是旧值 → 脏数据！且只有等到缓存过期才会恢复。
```

```
方案 B：先更新数据库，再删除缓存（推荐方案）

Thread-A: 更新数据库(新值) ✓
Thread-B: 读请求 → 缓存命中(旧值) → 返回旧值
Thread-A: 删除缓存 ✓

结果：Thread-B 读到一瞬间的旧值，但缓存已被删除，后续请求会读到新值。
      这是短暂的"最终一致性"窗口，在可接受范围内。
```

**但方案 B 也不是绝对安全的——极端情况下仍可能不一致：**

```
这个时序理论上可能（但概率极低）：

1. 缓存刚好过期（或被驱逐）
2. Thread-A 读 → 缓存 miss → 查库(旧值) → 正准备回填缓存时被挂起
3. Thread-B 写 → 更新数据库(新值) → 删除缓存（此时缓存本来就没有）
4. Thread-A 恢复 → 回填缓存(旧值)

结果：缓存是旧值，数据库是新值 → 脏数据
```

### 1.3 解决方案：延迟双删

```
func write(key, value):
    cache.delete(key)            // ① 先删一次
    db.update(key, value)        // ② 更新数据库
    sleep(几百毫秒)                // ③ 等待一段时间
    cache.delete(key)            // ④ 再删一次
```

**为什么延迟？** 给并发读请求留出"查库 + 回填缓存"的时间窗口，第二次删除把潜在的脏数据清掉。

**sleep 多长？** 一般是估算的"读请求查库 + 回填缓存"的耗时，或结合业务容忍度设置。

**更优雅的方案：订阅 binlog**
- 如果对一致性要求极高，可以用 Canal 等工具监听 MySQL binlog
- 当数据库更新后，binlog 触发异步删除缓存
- 配合消息队列的重试机制，保证删除最终成功

### 1.4 Cache-Aside 的优缺点

| 优点 | 缺点 |
|------|------|
| 实现简单，应用层控制，灵活性高 | 存在短暂不一致窗口 |
| 缓存只存热数据（懒加载），内存利用率高 | 首次访问必定 miss（缓存击穿风险） |
| 写操作直接删缓存，成本低 | 代码侵入性强，业务代码中散落缓存逻辑 |

**适用场景：** 读多写少，能容忍短暂不一致的绝大多数业务场景。这是 **最主流** 的策略。

## 2. Read/Write Through（读写穿透）—— 缓存层代理

### 2.1 核心思想

把缓存层从"应用负责"提升为"缓存层负责"——**应用只和缓存交互，缓存层自己负责与数据库同步**。

```
Cache-Aside 模式：               Read/Write Through 模式：

   App ──▶ Cache                    App ──▶ Cache（唯一入口）
    │        │                               │
    │        ▼                               ▼
    └──────▶ DB                          Cache ──▶ DB（缓存自己负责）
```

### 2.2 Read Through

```
func read(key):
    data = cache.get(key)
    if data != nil:
        return data
    
    // 缓存层自己去查数据库，应用无感知
    data = cache.loadFromDB(key)   // 同步加载
    return data
```

**与 Cache-Aside 的区别：**
- Cache-Aside 是应用去查 DB 再回填缓存
- Read Through 是缓存层封装了查 DB 的逻辑，应用只看到缓存接口

### 2.3 Write Through

```
func write(key, value):
    cache.set(key, value)          // ① 先写缓存
    cache.syncToDB(key, value)     // ② 同步写数据库（缓存层负责）
    return

// 整个过程对应用来说只是一个 cache.write() 调用
```

**特点：**
- 写操作是**同步**的——缓存和数据库同时更新后才返回成功。
- 写延迟 = 缓存写入延迟 + 数据库写入延迟（**比 Cache-Aside 慢**）。
- 数据一致性好——缓存和数据库始终保持同步。

### 2.4 优缺点

| 优点 | 缺点 |
|------|------|
| 应用代码简洁，无需在业务中维护缓存逻辑 | 写操作延迟高（同步写 DB + 缓存） |
| 缓存和数据库一致性更强 | 首次读同样 miss，缓存层需要实现 DB 加载逻辑 |
| 缓存总是最新数据（写穿透保证） | 缓存层必须存储全量数据（不能懒加载只存热的） |

**适用场景：** 需要强一致性，且对写入性能要求不高的场景。**注意：Redis 本身不支持 Read/Write Through**，需要应用层或中间件封装（如 Apache Ignite、Hazelcast 等，或自己用装饰器模式实现）。

**Redis 的局限性：** 从设计上来说 Redis 是一个单纯的缓存/存储，不提供内置的 DB 同步机制。要想在 Redis 上实现 Read/Write Through，需要自己写逻辑。

## 3. Write Behind / Write Back（写回）—— 异步批量写

### 3.1 核心思想

```
func write(key, value):
    cache.set(key, value)          // ① 只写缓存（立即返回）
    addToWriteQueue(key, value)    // ② 放入异步队列
    return success                 // ③ 立即返回

// 后台线程：
func flushWorker():
    while true:
        batch = collectBatch()
        db.batchUpdate(batch)      // 批量刷入数据库
```

**与 Write Through 的本质区别：**
- Write Through 是**同步**写 DB，Write Behind 是**异步**写 DB。
- 写延迟 = 缓存写入延迟（**极快**），DB 写入被异步化。

### 3.2 写合并（Write Coalescing）

```
假设在两次 flush 之间，同一个 key 被更新了 5 次：

Time:  t1→v1  t2→v2  t3→v3  t4→v4  t5→v5   flush
       ───────────────────────────────────▶
                   只写一次 v5

DB 实际只发生 1 次写入，而非 5 次 → 大幅降低 DB 写压力
```

### 3.3 优缺点

| 优点 | 缺点 |
|------|------|
| 写性能极高（只写缓存即返回） | **数据丢失风险**——缓存挂了，还没 flush 的数据全丢 |
| 写合并大幅减少 DB 压力 | 实现复杂——需要可靠的消息队列、重试、幂等 |
| 适合写密集型场景 | 数据一致性弱——DB 数据有明显滞后 |

**适用场景：** 写密集型、能容忍一定数据丢失的场景（如计数器、浏览数、点赞数等）。

### 3.4 Write Behind 在知名系统中的应用

Write Behind 并不是一个"只在理论中存在的模式"——它是**高性能系统设计的基石之一**，在数据库、操作系统、业务层都有广泛落地。

#### 3.4.1 MySQL InnoDB Buffer Pool

```
写请求：
           ┌────────────┐
  UPDATE ──▶ Buffer Pool│ (内存，修改页数据，标记为脏页)
           └──────┬─────┘
                  │ 写入 Redo Log (顺序写磁盘，极快)
                  │ 返回成功（不需要等数据页落地）
                  │
           ┌──────▼─────┐
           │ 数据页(磁盘) │  后台线程异步刷脏页
           └────────────┘  (Checkpoint / Page Cleaner 线程)
```

**关键机制：**
- 数据页修改在 **Buffer Pool（内存）** 中完成，修改后标记为**脏页（dirty page）**。
- 脏页由后台 **Page Cleaner 线程** 异步刷写到磁盘 `.ibd` 表空间。
- **Redo Log 保证持久性**：WAL（Write-Ahead Log）机制下，事务提交时先写 Redo Log（顺序写，很快），即使脏页还没刷盘，崩溃恢复时通过 Redo Log 重放数据。
- 脏页刷盘还支持**写合并**——同一页的多次修改在内存中合并为一次磁盘写入。

> **与 Write Behind 的对应关系：** Buffer Pool = 缓存层，数据文件 = 数据库，Redo Log = 持久化队列。这正是 Write Behind 的核心思想——先在内存（缓存）中完成写入，再异步批量持久化。

#### 3.4.2 操作系统页缓存（Page Cache）

```
应用写入文件：
  App ───write()──▶ Page Cache（内存）──▶ 返回"写入成功"
                          │
                          │ 后台：pdflush / kswapd 线程
                          │ 满足条件时（脏页占比超阈值 / 超时）
                          ▼
                    Disk（磁盘）

Linux 脏页回写触发条件（/proc/sys/vm/ 可调）：
  - dirty_background_ratio: 脏页占内存 10% 时开始后台回写
  - dirty_ratio: 脏页占内存 20% 时同步阻塞写入
  - dirty_expire_centisecs: 脏页超过 30s 必须回写
```

**关键机制：**
- 应用调用 `write()` 系统调用时，数据只是从用户态缓冲区拷贝到内核 **Page Cache**，然后立即返回。
- 内核后台线程（`pdflush`，新版为 `bdi-flush`）在脏页比例超过阈值或超时后，批量将脏页刷写到磁盘。
- 如果脏页占比达到 `dirty_ratio` 硬阈值，新的写入会被**同步阻塞**——这是操作系统的自我保护。

> **与 Write Behind 的对应关系：** Page Cache = 缓存层，应用写文件 = 业务写入，pdflush = 异步刷盘线程。文件系统的一切写入背后都是 Page Cache + Write Behind 机制。

#### 3.4.3 高频计数场景（浏览数、点赞数）

```
典型架构：

                   ┌──────────┐
  用户点赞 ────────▶  Redis   │ (INCR like_count:article_123)
                   └────┬─────┘
                        │ 内存中快速累加，响应 < 1ms
                        │
                   ┌────▼─────┐
                   │   MQ /   │  异步队列，削峰填谷
                   │ 定时任务  │
                   └────┬─────┘
                        │ 每 N 秒 / 每 M 条 批量同步
                        ▼
                   ┌──────────┐
                   │  MySQL   │  UPDATE article SET like_count = like_count + ?
                   └──────────┘
```

**为什么这个场景适合 Write Behind？**

| 特点 | 分析 |
|------|------|
| **写入极其频繁** | 热门文章可能每秒数千次点赞 / 阅读，直接写 DB 扛不住 |
| **允许短暂不一致** | 用户不关心点赞数精确到秒，展示时差几十个不影响体验 |
| **可以批量合并** | 一个文章在 5 秒内的 500 次点赞，合并成 1 次 `UPDATE SET like_count = like_count + 500` |
| **少量丢失可接受** | Redis 极小概率宕机丢几秒数据，对业务无实质影响（相比性能收益） |

**实现要点：**
1. Redis 中使用 `INCR` / `HINCRBY` 做原子累加。
2. 定时任务（如每 5 秒）扫描 Redis 中的计数器，批量 `UPDATE` 到 MySQL。
3. 可选引入 MQ 解耦——Redis 达到一定阈值后写 MQ，消费者批量落库。
4. 监控 Redis 内存增长，避免计数器 key 无限膨胀。

> **本质洞察：** Write Behind 在这类场景中不仅是性能优化，更是**架构层面的流量缓冲器**——用 Redis 内存性能吸收峰值写入，再用批量落库保护数据库，实现"写的压力不上移到 DB"。

## ⭐️ 三种策略对比总结

| 维度 | Cache-Aside | Read/Write Through | Write Behind |
|------|------------|-------------------|--------------|
| **核心思想** | 应用控制缓存与 DB | 缓存层代理 DB 访问 | 异步批量刷 DB |
| **读延迟** | 命中低 / miss 高 | 命中低 / miss 高 | 同 Cache-Aside |
| **写延迟** | 低（只删缓存+写DB） | 高（同步写缓存+DB） | 极低（只写缓存） |
| **一致性** | 最终一致，有短暂窗口 | 较强 | 弱，DB 滞后明显 |
| **数据丢失风险** | 低（DB 是先写的） | 低（同步写 DB） | **高**（缓存故障丢失） |
| **实现复杂度** | 低 | 中（需封装缓存层） | **高**（队列+重试+幂等） |
| **Redis 原生支持** | ✅ 应用层实现 | ❌ 需自己封装 | ❌ 需自己实现 |

## ⭐️ 面试题汇总

### Q1: 为什么 Cache-Aside 是先更新 DB 后删除缓存？

> 主要为了防止并发场景下的脏数据。如果先删缓存后更新 DB，在"删缓存→更新 DB"的窗口内，并发读请求可能查到 DB 旧值并回填缓存，导致缓存出现永不过期的脏数据。而先更新 DB 后删缓存，最坏情况下只会出现一次短暂的旧值读取，缓存删除后就会恢复。

### Q2: Cache-Aside 一定能保证最终一致吗？有没有极端情况？

> 不能绝对保证。存在一个理论上可能的极端时序：缓存过期 → 读请求查库（旧值）→ 写请求更新 DB（新值）→ 写请求删缓存（但缓存本来就无）→ 读请求回填缓存（旧值）。此时 DB 是新值，缓存是旧值。解决方案有延迟双删、订阅 binlog 异步删缓存等。

### Q3: Write Through 和 Write Behind 有什么区别？各自适用什么场景？

> **Write Through 是同步写**——缓存和 DB 同时更新后才返回，一致性强但延迟高，适用于对一致性要求高的场景（如金融）。
> **Write Behind 是异步写**——只写缓存立即返回，后台批量刷 DB，写性能极高但有数据丢失风险，适用于写密集且允许少量丢失的场景（如计数、日志）。

### Q4: 为什么 Redis 默认不支持 Read/Write Through？

> Redis 定位是**缓存/存储中间件**，而非数据库代理。它没有内置的数据库同步机制。社区方案通常是在应用层用装饰器模式封装，或借助 Redisson 等客户端库来实现类似效果。

### Q5: 三种策略如何选型？

> - **绝大多数业务**：Cache-Aside，简单灵活，最终一致性可接受。
> - **强一致性、写不多**：Read/Write Through（需自己封装实现）。
> - **写密集型、允许少量丢失**：Write Behind（如秒杀库存扣减、点赞计数）。
