# 秒杀系统设计 —— 从超卖到生产级的完整演进

---

## 前言：这篇文章在讲什么？

秒杀是面试中考察**综合工程能力**的经典题目。它不是一个单纯的 Redis 问题——它涉及到并发控制、库存扣减、分布式锁、消息队列、性能优化等多个领域。

本文用一个秒杀系统从 0 到 1 的演进过程，把每个阶段的**代码、问题、解决方案**都讲清楚。

```
演进路线：
  V1: 无锁扣库存 → 超卖！
  V2: synchronized → 单机有效，分布式无效
  V3: Redis 分布式锁 → 解决分布式，但性能差
  V4: Redis 原子扣库存 → 高性能 + 一人一单
  V5: 异步秒杀 → 削峰填谷，保护后端
  V6: 生产级架构 → 前端限流 + Redis 预热 + Lua 扣库存 + MQ 异步下单
```

## 第一章：什么是超卖？为什么会发生？

### 1.1 超卖的场景

```
一个秒杀活动：iPhone 15，库存 100 台，价格 1 元
参与用户：100 万人同时抢购

期望结果：只有前 100 个用户能买到
实际结果：因为并发问题，卖了 150 台 → 超卖 50 台！
```

### 1.2 超卖的根本原因——"读和写之间被人插队了"

```java
// 典型的扣库存代码（有并发漏洞）
public boolean buy(Long productId) {
    // ① 查库存
    int stock = productMapper.selectStock(productId);  // → 读到 stock = 1

    // ② 判断库存
    if (stock <= 0) {
        return false;  // 库存不足
    }

    // ③ 扣库存
    productMapper.updateStock(productId, stock - 1);  // → UPDATE SET stock = 0
    return true;
}
```

```
为什么超卖？看两条并发请求的时间线：

  线程 A                          线程 B
  ────────                        ────────
  ① 查库存 → stock = 1
                                  ① 查库存 → stock = 1  ← 也读到了 1！
  ② 判断：stock > 0 ✓
                                  ② 判断：stock > 0 ✓   ← 两个线程都通过了检查！
  ③ 扣库存 → stock = 0
                                  ③ 扣库存 → stock = 0   ← 扣了两次！

  结果：库存减了 2 次，但实际只有 1 件商品 → 超卖！

根本矛盾：
  线程 A 的"查库存"和"扣库存"不是原子操作
  线程 B 在 A 还未扣完时插入进来，读到了"脏数据"
```

**这就是典型的 "check-then-act" 竞态条件——Java 并发的经典问题，在数据库层面同样存在。**

## 第二章：V1—V2 从单机锁到分布式锁

### 2.1 V1：synchronized —— 单机有效，集群失效

```java
/**
 * V1：用 synchronized 锁住整个方法
 * 问题：只能锁住当前 JVM，管不了其他服务器
 */
public synchronized boolean buyV1(Long productId, Long userId) {
    int stock = productMapper.selectStock(productId);
    if (stock <= 0) return false;
    productMapper.updateStock(productId, stock - 1);
    // 创建订单...
    return true;
}
```

```
单机部署（1 台服务器）：
  synchronized → 锁住当前 JVM 内的所有线程 → 有效 ✓

集群部署（3 台服务器）：
  ┌────────────┐  ┌────────────┐  ┌────────────┐
  │  服务器 A   │  │  服务器 B   │  │  服务器 C   │
  │synchronized│  │synchronized│  │synchronized│  ← 三把独立的锁！
  └─────┬──────┘  └─────┬──────┘  └─────┬──────┘
        └────────────────┼───────────────┘
                         ▼
                  ┌─────────────┐
                  │   MySQL     │  ← 三个线程同时查库存 → 超卖！
                  └─────────────┘
```

### 2.2 V2：Redis 分布式锁 —— 解决分布式互斥

```java
/**
 * V2：用 Redis 分布式锁保证跨服务器的互斥
 */
public boolean buyV2(Long productId, Long userId) {
    String lockKey = "lock:product:" + productId;
    String lockValue = UUID.randomUUID() + ":" + Thread.currentThread().getName();

    // ① 获取分布式锁
    boolean gotLock = "OK".equals(
        jedis.set(lockKey, lockValue, SetParams.setParams().nx().px(3000))
    );
    if (!gotLock) {
        return false;  // 没拿到锁 → 别人在处理
    }

    try {
        // ② 查库存 + 扣库存（临界区）
        int stock = productMapper.selectStock(productId);
        if (stock <= 0) return false;
        productMapper.updateStock(productId, stock - 1);
        // ③ 创建订单...
        return true;
    } finally {
        // ④ 释放锁（Lua 验证身份）
        String lua = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                     "return redis.call('del', KEYS[1]) else return 0 end";
        jedis.eval(lua,
            Collections.singletonList(lockKey),
            Collections.singletonList(lockValue));
    }
}
```

**V2 的问题：性能瓶颈。**

```
100 万人抢 100 件商品：
  只有 100 个人能拿到锁 → 但 999900 个人也在排队等锁！
  每个人都要"拿锁 → 查 DB → 发现库存 0 → 释放锁"
  即使库存已经为 0 了，后续的 99 万人还是要走一遍"拿锁查 DB"的流程
  这一过程涉及到 100 万次 Redis 锁 + 100 万次 DB 查询
  DB 被反复无效查询打爆
```

## 第三章：V3——先判断库存，再拿锁

### 3.1 改进思路

**不要把锁的粒度搞太大了。** 在拿锁之前先做一次快速库存判断——如果库存已经没了，连锁都不用拿。

```java
/**
 * V3：先做一次轻量的库存检查，再拿锁
 */
public boolean buyV3(Long productId, Long userId) {
    // ① 先查 Redis 中的库存（轻量，不用拿锁）
    String stockKey = "stock:product:" + productId;
    String stockStr = jedis.get(stockKey);
    if (stockStr == null || Integer.parseInt(stockStr) <= 0) {
        return false;  // 库存为 0 → 直接返回，不拿锁了！
    }

    // ② 库存 > 0 → 再拿锁
    String lockKey = "lock:product:" + productId;
    String lockValue = UUID.randomUUID() + ":" + Thread.currentThread().getName();
    boolean gotLock = "OK".equals(
        jedis.set(lockKey, lockValue, SetParams.setParams().nx().px(3000))
    );
    if (!gotLock) return false;

    try {
        // ③ 再次确认库存（Double Check）
        int stock = productMapper.selectStock(productId);
        if (stock <= 0) return false;

        // ④ 扣库存
        productMapper.updateStock(productId, stock - 1);
        // ⑤ 同步更新 Redis 库存
        jedis.set(stockKey, String.valueOf(stock - 1));
        return true;
    } finally {
        String lua = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                     "return redis.call('del', KEYS[1]) else return 0 end";
        jedis.eval(lua,
            Collections.singletonList(lockKey),
            Collections.singletonList(lockValue));
    }
}
```

**V3 的提升：** 库存为 0 后，后续请求在步骤 ① 就被拦住了（一次 Redis GET），不再拿锁、不再查 DB。

**但 V3 仍不够——Redis 库存和 DB 库存有同步问题、每次扣库存还是要查 DB。**

## 第四章：V4——Redis 原子扣库存 + Lua

### 4.1 核心思路：库存完全由 Redis 管理

**Redis 单线程执行命令，天然适合做原子库存扣减。** 用 Lua 脚本把"判断库存 + 扣库存"打包成一个原子操作：

```java
/**
 * V4：Redis Lua 原子扣库存 —— 高性能、不超卖
 *
 * Redis 中预存库存：SET stock:product:1001 100
 */
public class SeckillServiceV4 {
    private final Jedis jedis;

    /**
     * 扣库存 Lua 脚本
     * KEYS[1] = 库存 key，如 "stock:product:1001"
     * ARGV[1] = 用户 ID（用于一人一单校验）
     * ARGV[2] = 购买的套数
     * 返回：0=成功, -1=库存不足, -2=已购买过
     */

    // ═══════════ Lua 1：纯扣库存 ═══════════
    private static final String DEDUCT_STOCK_LUA =
        "local stock = tonumber(redis.call('get', KEYS[1]) or 0) " +  // ① 读当前库存
        "if stock <= 0 then " +
        "    return -1 " +                                               // ② 库存不足
        "end " +
        "redis.call('decr', KEYS[1]) " +                                // ③ DECR 扣库存
        "return 0";                                                      // ④ 成功

    // ═══════════ Lua 2：扣库存 + 一人一单 ═══════════
    // KEYS[1] = 库存 key
    // KEYS[2] = 已购买用户 Set
    // ARGV[1] = 用户 ID
    private static final String DEDUCT_STOCK_ONE_PER_USER_LUA =
        // ① 检查是否已购买（SISMEMBER O(1)）
        "local bought = redis.call('SISMEMBER', KEYS[2], ARGV[1]) " +
        "if bought == 1 then " +
        "    return -2 " +                                               // 已购买过
        "end " +

        // ② 检查库存
        "local stock = tonumber(redis.call('get', KEYS[1]) or 0) " +
        "if stock <= 0 then " +
        "    return -1 " +                                               // 库存不足
        "end " +

        // ③ 扣库存 + 记录用户
        "redis.call('decr', KEYS[1]) " +                                 // DECR stock
        "redis.call('sadd', KEYS[2], ARGV[1]) " +                       // SADD users:product:1001 userId
        "return 0";                                                      // 成功

    /**
     * 纯扣库存（不限购）
     */
    public int deductStock(Long productId) {
        String stockKey = "stock:product:" + productId;
        Object result = jedis.eval(DEDUCT_STOCK_LUA,
            Collections.singletonList(stockKey),
            Collections.emptyList());
        return ((Long) result).intValue();  // 0=成功, -1=库存不足
    }

    /**
     * 扣库存 + 一人一单（防黄牛）
     */
    public int deductStockOnePerUser(Long productId, Long userId) {
        String stockKey = "stock:product:" + productId;
        String usersKey = "users:product:" + productId;

        Object result = jedis.eval(DEDUCT_STOCK_ONE_PER_USER_LUA,
            Arrays.asList(stockKey, usersKey),
            Collections.singletonList(String.valueOf(userId)));
        return ((Long) result).intValue();  // 0=成功, -1=库存不足, -2=已购买过
    }
}
```

**执行效果：**

```bash
# 秒杀开始前初始化
SET stock:product:1001 100                    # 库存 100
DEL users:product:1001                        # 清空已购用户

# 用户 10086 秒杀
EVAL "..." 2 stock:product:1001 users:product:1001 10086
→ 0  # 成功！库存 = 99，users Set 包含了 10086

# 用户 10086 第二次秒杀
EVAL "..." 2 stock:product:1001 users:product:1001 10086
→ -2  # 已购买过，拒绝！

# 100 个人购买后
GET stock:product:1001 → 0
# 第 101 个用户
EVAL "..." 2 stock:product:1001 users:product:1001 10087
→ -1  # 库存不足
```

### 4.2 一人一单——并发安全分析

#### 同一用户重复点击的拦截流程

```
═══════════════════════════════════════════════════════════════
     一人一单：Lua 原子拦截重复购买（userId=10086 连点两次）
═══════════════════════════════════════════════════════════════

  第一次点击（t=0ms）：
    ┌─────────────────────────────────────┐
    │ Lua:                                │
    │ ① SISMEMBER users:prod:1001 10086  │ → 0 (没买过)
    │ ② GET stock:prod:1001              │ → 100
    │ ③ DECR stock:prod:1001             │ → 99
    │ ④ SADD users:prod:1001 10086       │ → 加入 Set
    │ ⑤ return 0                         │
    └─────────────────────────────────────┘
    → 用户看到：抢购成功！

  第二次点击（t=50ms，用户手抖又点了一次）：
    ┌─────────────────────────────────────┐
    │ Lua:                                │
    │ ① SISMEMBER users:prod:1001 10086  │ → 1 (已经买过了!)
    │ ② return -2  ← 直接返回，不走③④⑤    │
    └─────────────────────────────────────┘
    → 用户看到：不能重复购买

  关键：SISMEMBER 和 SADD 在同一个 Lua 脚本中原子执行
        不可能出现"SISMEMBER 返回 0 但另一个线程已经 SADD 了"的情况
```

#### Set vs 布隆过滤器

```
为什么用 Set 而不是用 bloom filter？

  方案 A：SISMEMBER users:product:1001 userId  ← Set，O(1)，100% 准确
        内存占用：100 万用户 × ~50B ≈ 50MB

  方案 B：布隆过滤器 → O(1)，1% 误判，12KB
        1% 误判 → 100 万个用户中有 1 万人被冤枉 → 投诉暴增
  
  结论：秒杀用 Set！用户"不能买到"已经是坏体验了，
       绝不能再被布隆过滤器误判。
       只有"预过滤海量不存在用户"才用布隆过滤器，比如拦截攻击流量。
```

### 4.3 V4 的性能分析

```
Lua 脚本在 Redis 服务端原子执行：
  GET stock → 判断 → DECR → SADD → 全部在一个调用中完成
  
  单次执行时间：~10-30 微秒（因为全是内存操作，无网络 RTT 往返）
  单机 QPS：约 3-5 万/s

对比 V2 的分布式锁方案：
  V2：拿锁(1次RTT) → 查DB → 扣DB(1次RTT) → 更新Redis(1次RTT) → 释放锁(1次RTT)
      至少 3-4 次网络 RTT + 1 次 DB 磁盘 I/O
  V4：1 次 EVAL → 1 次网络 RTT
      快了 3-4 倍，且完全不碰 DB
```

## 第五章：V5——异步秒杀（消息队列削峰）

### 5.1 V4 的问题：虽然扣库存快，但下单仍然慢

```
V4 的秒杀流程：

  用户请求 → Redis Lua 扣库存（极快，~20μs）
           → 生成订单（慢！要 INSERT 到 MySQL，可能 10-50ms）
           → 扣减账户余额 / 积分
           → 发送通知
           → 返回结果给用户

  问题：扣库存只要 20μs，但生成订单要 50ms
        1 秒内可以扣 50000 次库存，但只能生成 20 个订单（数据库限制）
        → 用户"抢到了"但等半天才能看到结果
```

### 5.2 异步化——扣库存和下单解耦

```
V5 架构：

  ┌─────────────────────────────────────────────────────────┐
```

```java
@RestController
@RequestMapping("/seckill")
public class SeckillControllerV5 {
    @Autowired private StringRedisTemplate redisTemplate;
    @Autowired private SeckillOrderConsumer orderConsumer;  // 下单消费者

    // ═══════════ Lua：扣库存 + 写消息 ═══════════
    // KEYS[1] = stock:product:{id}
    // KEYS[2] = users:product:{id}
    // KEYS[3] = stream:orders          ← Stream key
    // ARGV[1] = userId
    // ARGV[2] = productId
    // ARGV[3] = 订单消息 JSON（productId + userId + timestamp）
    private static final String SECKILL_LUA =
        // ① 一人一单检查
        "local bought = redis.call('SISMEMBER', KEYS[2], ARGV[1]) " +
        "if bought == 1 then return -2 end " +

        // ② 库存检查
        "local stock = tonumber(redis.call('get', KEYS[1]) or 0) " +
        "if stock <= 0 then return -1 end " +

        // ③ 扣库存
        "redis.call('decr', KEYS[1]) " +
        // ④ 记录用户
        "redis.call('sadd', KEYS[2], ARGV[1]) " +
        // ⑤ 写入 Stream 消息队列（XADD stream * productId xxx userId xxx）
        "redis.call('xadd', KEYS[3], '*', " +
        "    'productId', ARGV[2], 'userId', ARGV[1], 'timestamp', ARGV[3]) " +
        "return 0";

    /**
     * 秒杀接口——只做两件事：① 扣库存 ② 写入消息队列
     * 返回结果：0=成功, -1=库存不足, -2=重复购买
     */
    @PostMapping("/{productId}")
    public Result seckill(@PathVariable Long productId, HttpServletRequest request) {
        Long userId = getCurrentUserId(request);

        // ① 执行 Lua：扣库存 + 一人一单 + 写入 Stream
        String stockKey = "stock:product:" + productId;
        String usersKey = "users:product:" + productId;
        String streamKey = "stream:orders";

        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(SECKILL_LUA, Long.class),
            Arrays.asList(stockKey, usersKey, streamKey),
            String.valueOf(userId),
            String.valueOf(productId),
            String.valueOf(System.currentTimeMillis())
        );

        // ② 根据返回值告知用户
        if (result == 0) {
            return Result.ok("抢购成功，订单生成中...");  // 用户立即得到反馈
        } else if (result == -1) {
            return Result.fail("已售罄");
        } else {
            return Result.fail("请勿重复购买");
        }
    }
}
```

### 6.3 库存预热

```java
/**
 * 秒杀开始前，将库存加载到 Redis
 * 在秒杀开始前的某个时刻执行（如 12:00:00 秒杀，11:59:50 执行）
 */
@Component
public class StockPreheater {

    @Autowired private StringRedisTemplate redisTemplate;
    @Autowired private ProductMapper productMapper;

    /**
     * 秒杀前的库存预热（由定时任务或手动接口触发）
     */
    public void preheat(Long productId) {
        // ① 从 DB 加载库存
        int dbStock = productMapper.selectStock(productId);

        // ② 写入 Redis
        redisTemplate.opsForValue()
            .set("stock:product:" + productId, String.valueOf(dbStock));

        // ③ 清空已购买用户的 Set（新秒杀开始时应该是空的）
        redisTemplate.delete("users:product:" + productId);

        // ④ 可选：把商品标记为"秒杀中"状态
        redisTemplate.opsForValue()
            .set("seckill:status:" + productId, "ACTIVE");
    }
}
```

### 6.5 秒杀系统的异常处理

```
═══════════════════════════════════════════════════════════════
              五种异常场景的检测与恢复路径
═══════════════════════════════════════════════════════════════

场景 1：Redis 扣库存成功，但写 Stream 失败
  ┌─────────────────────────────────────────┐
  │ Lua 脚本中 DECR 和 XADD 在同一个脚本    │
  │ Redis 单线程执行 Lua → 要么全成功要么全失败 │
  │ 不会出现"库存扣了但消息没写进去"的情况     │
  └─────────────────────────────────────────┘
  恢复：不需要——Lua 原子保证了

场景 2：Stream 写入成功，消费者拿消息后下单前崩溃
  ┌─────────────────────────────────────────┐
  │ msg-001 在 PEL 中，状态=PENDING          │
  │ 消费者崩溃 → 消息 idle 时间持续增长       │
  │ 10 秒后 → 另一个消费者 XAUTOCLAIM 认领   │
  │ 重新尝试下单 → 下单前 SISMEMBER 幂等判断 │
  └─────────────────────────────────────────┘
  恢复：XAUTOCLAIM 自动认领 → 重新下单 → 幂等防护

场景 3：下单成功但 ACK 失败
  ┌─────────────────────────────────────────┐
  │ msg-001 已在 DB 中有订单记录             │
  │ 但 PEL 中仍存留 → 被重新认领 → 重新下单  │
  │ 幂等判断：SELECT COUNT(*) WHERE          │
  │   product_id=? AND user_id=?            │
  │ → 已存在 → 直接 ACK，不重复下单          │
  └─────────────────────────────────────────┘
  恢复：幂等校验 → 直接 ACK → 不重复下单

场景 4：Redis 库存和 DB 订单数不一致
  ┌─────────────────────────────────────────┐
  │ 实际售出 = Redis Set 中的用户数(可靠)    │
  │ DB 订单数 ≠ Redis Set 用户数             │
  │ ↓                                       │
  │ 定时核对任务扫描差异                      │
  │ ↓                                       │
  │ DB 少：补偿创建订单                      │
  │ DB 多：标记异常订单(人工处理)              │
  └─────────────────────────────────────────┘
  恢复：定时核对 + 补偿任务

场景 5：秒杀过程中 Redis 主库挂了
  ┌─────────────────────────────────────────┐
  │ Sentinel 检测 Master 宕机 (5-15秒)       │
  │ → 选举新 Master (Slave)                 │
  │ → 新 Master 库存 = 异步复制时刻的快照     │
  │   (可能比实际多几个，因为 Master 宕机前   │
  │    最后一次修改可能还没同步到 Slave)      │
  │ ↓                                       │
  │ 秒杀结束后：以 DB 最终订单数为准          │
  │ 如果 Redis 库存 > DB 订单数 → 修正       │
  └─────────────────────────────────────────┘
  恢复：主从切换 + 结束后数据核对修正
```

## ⭐️ 面试题汇总

**Q1: 如何解决超卖问题？**

> 超卖的根本原因是"查库存"和"扣库存"不是原子操作。解决方案：① Redis Lua 原子扣库存（DECR 在单线程中执行）；② DB 层面加乐观锁（`UPDATE stock SET stock = stock - 1 WHERE stock > 0`）；③ 分布式锁（性能差，不推荐秒杀场景）。最优方案：Redis Lua + 异步下单。

**Q2: 一人一单怎么实现？**

> 用 Redis Set 记录已购买的用户。每次秒杀时，Lua 脚本先 `SISMEMBER` 检查用户是否已购 → 否 → 继续扣库存并 `SADD` 加入 Set。Set 的 O(1) 查找保证性能。如果 Set 过大（上亿用户），秒杀结束后按产品过期清理即可。

**Q3: 为什么秒杀要用异步而不是同步？**

> 同步：扣库存（微秒级）和下单（几十毫秒级）混在一条处理链路中，下单速度卡住整个系统。异步：扣库存后立即返回结果给用户（30μs），把下单任务写入 MQ 后台慢慢消费。用户体验从"等 50ms 可能失败"变成"瞬间知道结果"；DB 可以按照自己舒服的速度生成订单。

**Q4: Redis 减了库存但下单失败了，怎么补偿？**

> ① 写入 Stream 和扣库存在同一个 Lua 中——写 MQ 失败 = 库存不扣，原子保证；② 消费者处理失败 → 消息留在 PEL 不 ACK → 定时重试；③ 最终还是失败 → 人工补偿脚本（INCR 恢复库存 + SREM 移除用户），或由定时核对程序自动处理。

**Q5: 秒杀系统从哪些层面做限流？**

> ① Nginx 层：IP 限流 + 验证码（挡住脚本）；② Gateway 层：令牌桶全局限流（5000 QPS）；③ Redis 层：Lua 库存判断（库存=0 后直接返回，不再消耗后端资源）；④ DB 层：消费者限速（每秒最多处理 100 个订单）。层层拦截，保证后端不被打挂。

**Q6: 秒杀结束后 Redis 和 DB 库存不一致怎么办？**

> 秒杀关闭后执行数据核对：实际售出 = Redis Set 中的用户数（这个和 Redis 扣库存过程一致）。用这个数字和 DB 订单数比对 → 不一致则触发补偿：多扣的库存 INCR 恢复，多产生的订单标记异常。最终以 DB 中的数据为准。
