# Redis 内存管理 —— 过期与淘汰机制深度剖解

---

## 前言：Redis 为什么需要内存管理？:rocket::rocket::rocket:

Redis 是**内存数据库**——所有数据都在内存中。内存是有限资源：一台服务器 64GB 内存，你不能把全公司的所有数据都塞进去。

```
内存管理的两个维度：

  维度 1：主动过期
    由"用户设置 TTL"驱动 → "这个 key 应该什么时候被删除"
    适用：临时数据（验证码、Session、缓存）


  维度 2：被动淘汰
    由"内存不够用"驱动 → "内存满了，新数据进不来，要删谁"
    适用：所有数据都可能被淘汰的风险管控
```

**本文将分别展开这两个维度。**

---

## 第一章：为什么要给缓存数据设过期时间？

### 1.1 不设过期时间会怎样？

```bash
# 场景：商品缓存
SET product:1001 '{"name":"iPhone","price":9999}'
# 没有 EXPIRE → 这个 key 永远存在

# 商品下架了，数据库里这条记录已删除
# 但 Redis 中的缓存还在！用户永远看到旧数据！
# → 缓存和数据库不一致
```

```
不设过期导致的两个问题：
  ① 脏数据——数据源已经变了/删了，Redis 里还是旧版本
  ② 内存泄漏——不断 SET 新 key 但不删旧的 → 内存无限增长 → OOM
```

### 1.2 过期时间的妙用

```bash
# 验证码（业务天然过期） 
SETEX sms:13800138000 300 "123456"   # 5 分钟后自动删除

# 分布式锁（锁自动释放） 
SET lock:order NX PX 30000          # 30 秒后自动释放

# Session（用户不活跃自动退出）
SETEX session:abc123 1800 ""         # 30 分钟无操作自动退出

# 热点数据（有过期时间兜底）
SETEX hot:article:123 3600 ""        # 1 小时后自动刷新
```

## 第二章：Redis 如何判断一个 key 是否过期？

### 2.1 过期字典——expires dict:rocket::rocket::rocket::rofl::rofl::rofl:

Redis 内部**不把过期时间存在 key 本身上**，而是单独维护了一张"过期字典"（expires dict）。

```
Redis 内部的两套结构：

  键空间 (key space dict)：
    存所有的 key → value 映射
    "product:1001" → '{"name":"iPhone"}'
    "counter" → "5000"
    "sms:13800138000" → "123456"

  过期字典 (expires dict)：
    只存"设置了 TTL 的 key" → 过期时间（Unix 毫秒时间戳）
    "product:1001" → 1715000000000
    "sms:13800138000" → 1715000300000
    （注意："counter" 没有 TTL，不出现在这个字典里）
```

```c
// expires dict 中的一个 entry 长这样：
// key   = 指向和键空间同一个 SDS 对象的指针（共享，不额外占用内存）
// value = 过期时间戳（Unix 毫秒）
```

### 2.2 判断过期的内部逻辑:rocket::rofl:

```
判断一个 key 是否过期：
  ① 在过期字典中查找该 key
  ② 如果不存在 → 没设 TTL → 永不过期
  ③ 如果存在 → 取出过期时间戳
  ④ 比较：当前系统时间 ≥ 过期时间戳 → 已过期
```

**关键：** 这个判断是 O(1) 的——直接查 dict。

## 第三章：过期 key 是怎么被删除的？——三种策略

### 3.1 策略一：惰性删除（Lazy Expire）:rocket::rofl::rofl::rofl::rofl::rofl:

**思路：** 不主动找，等有人访问这个 key 时再检查。如果过期了，顺手删掉。

```
惰性删除的触发时机：
  SET / GET / INCR / HGET / ... 任何对 key 的访问时

流程：
  ① 客户端请求 GET product:1001
  ② Redis 查键空间 → 找到了 value
  ③ Redis 查过期字典 → 有 TTL → 检查是否过期
     ├─ 未过期 → 返回 value
     └─ 已过期 → 删除 key（同步），返回 nil
```

**优点：** CPU 友好——不会主动扫描，只在访问时才检查。

**缺点：** 内存不友好——有一个 key 过期后再也没人访问它，那它就永远占着内存。:rofl::rofl::rofl:

### 3.2 策略二：定期删除（Active Expire）

**思路：** 每隔一段时间，随机抽查一部分 key:rofl::rofl:，发现过期的删掉。

```
定期删除的执行入口：
  Redis 的 serverCron 定时任务（每 100ms 触发一次）
  
  每次执行：
    ① 从 expires dict 中随机抽取 20 个 key
    ② 检查这 20 个中是否过期 → 过期的 DEL 掉
    ③ 如果过期比例 > 25%（≥5 个过期） → 重复步骤①②
    ④ 但每次最多执行 25ms → 到了时间就停下
       （不能占主线程太久，避免阻塞客户端请求）

  为什么是 25ms？
    100ms 的 serverCron 周期中，留 25ms 给过期 key 清理，
    超过这个时间必须交出 CPU，让主线程处理其他任务。
```

**这个算法很巧妙——"抽样的 + 限时的"：**

```
随机抽样：避免遍历所有 key（O(N) 可能秒级阻塞）
限时 25ms：过期 key 再多，也不会占满主线程

最坏情况：过期 key 特别多，每次 25ms 也清不完
         → 剩下的过期 key 留在内存中
         → 下一次 100ms 周期继续清
         → 直到清理完为止
```

### 3.3 惰性 + 定期 = 完美配合:rofl::rofl::rofl::rofl::rofl:

```
为什么同时需要两种策略？

  只有惰性：没人访问的过期 key 永久占内存 → 内存泄漏
  只有定期：expires dict 有几百万 key 时，随机抽样可能总是抽不到过期的 → 清理效率低

  两者配合：
    定期删除 → 保证过期 key 最终会被清掉（即使没人访问）
    惰性删除 → 保证被访问的过期 key 立即清掉（不影响正确性）
```

## 第四章：大量 key 集中过期怎么办？

### 4.1 问题现象

```
场景：一个缓存预热脚本在凌晨 0 点批量加载了 100 万个 key
      每个 key 的 TTL 都设为 3600 秒

      凌晨 1 点 → 100 万个 key 同时过期！

      Redis 定期删除（每 100ms 一次）开始处理：
        → 随机抽 20 个 → 全过期（100%） → 再来一轮
        → 再随机抽 20 个 → 全过期 → 再来一轮
        → ... 一直循环，直到过了 25ms
        → 25ms 内清理了 5000 个 key，还剩 995000 个
        → 下一轮 100ms → 继续清 25ms → 一直循环
        
      在此期间 → 定期删除占用主线程 25% 的 CPU 时间
      如果访问这些过期 key 的请求很多 → 惰性删除也频繁触发
      → 综合效果：Redis 响应变慢 → 延迟抖动
```

### 4.2 解决方案

**方案一：TTL 加随机偏移（同缓存雪崩预防）**

```java
// 基础 TTL = 3600 秒 + 随机偏移 0~1080 秒
long ttl = 3600 + (long) (Math.random() * 3600 * 0.3);
jedis.setex(key, ttl, value);
// 100 万个 key 分散在 3600~4680 秒间 → 不会同时过期
```

**方案二：程序层面错峰**

```java
// 不要把 100 万个 key 一次性加载，分批
for (int i = 0; i < 1000000; i += 10000) {
    batchLoad(i, 10000);  // 每次加载 1 万个
    Thread.sleep(1000);    // 隔 1 秒再加载下一批
}
```

## 第五章：内存淘汰策略——内存满了怎么办？

### 5.1 场景:rofl:

```
Redis 的 maxmemory 设置为 2GB
当前内存使用：2.1GB（超过了 maxmemory）

这时来了一个新请求：SET new_key "value"
Redis 该怎么办？
  → 拒绝写入？→ 用户操作失败
  → 删掉一个旧 key 腾地方？→ 删哪个？
```

这就是**内存淘汰策略**要回答的问题。

### 5.2 八种淘汰策略一览

Redis 提供了 8 种策略，分成三大类：:rofl::rofl::rofl::rofl::rofl::

```
┌─────────────────────────────────────────────────────────────────┐
│  Redis 内存淘汰策略（8 种，注解 maxmemory-policy）                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【不淘汰】                                                      │
│  noeviction：不淘汰任何 key，写操作直接报错（默认策略）            │
│                                                                 │
│  【针对"所有 key"的淘汰——allkeys】                               │
│  allkeys-lru：在所有 key 中，淘汰"最近最少使用"的（LRU）           │
│  allkeys-lfu：在所有 key 中，淘汰"最不经常使用"的（LFU）           │
│  allkeys-random：在所有 key 中，随机淘汰                          │
│                                                                 │
│  【针对"设置了 TTL 的 key"的淘汰——volatile】                     │
│  volatile-lru：在有 TTL 的 key 中，淘汰"最近最少使用"的            │
│  volatile-lfu：在有 TTL 的 key 中，淘汰"最不经常使用"的            │
│  volatile-random：在有 TTL 的 key 中，随机淘汰                    │
│  volatile-ttl：在有 TTL 的 key 中，淘汰"最快过期"的                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3 allkeys vs volatile——什么时候选哪个:rofl::rofl::rofl:

```
allkeys-xxx：缓存场景（推荐）
  所有 key 都是缓存，没有"不能删"的数据
  淘汰范围大 → 更容易找到"不值得保留"的 key
  如：allkeys-lru 是缓存场景的最佳实践

volatile-xxx：混合场景——部分数据"永久保存"，部分"临时缓存"
  有些 key 没有 TTL（视为持久数据，不能被淘汰）
  有些 key 有 TTL（缓存数据，可以被淘汰）
  淘汰范围只限制在有 TTL 的 key

noeviction：不能丢数据的场景
  数据比内存还重要——宁愿写失败，不丢任何一条
  默认策略——不配置 maxmemory-policy 就是 noeviction
```

### 5.4 LRU vs LFU——"最近" vs "经常"

这两个概念经常被混淆：

```
LRU（Least Recently Used——最近最少使用）：

  淘汰思路：谁最久没被访问，就淘汰谁
  衡量：时间距离（多久没被访问了）

  例子：
    用户 A：昨天访问了一次（距今 24 小时）
    用户 B：今天上午访问了一次（距今 4 小时）
    → LRU 淘汰用户 A（A 更久没被访问）


LFU（Least Frequently Used——最不经常使用）：

  淘汰思路：谁被访问的次数最少，就淘汰谁
  衡量：频率（单位时间内被访问了多少次）

  例子：
    用户 A：一天访问了 100 次（频繁访问）
    用户 B：一天只访问了 3 次（偶尔访问）
    → LFU 淘汰用户 B（B 虽然最近访问过，但历史访问频率低）
```

**什么时候用 LFU 而不是 LRU？**:rocket::rofl::rofl::rofl::rofl:

```
LRU 的问题：
  一个冷门 key 刚刚被偶然访问了一次 → LRU 认为它是"热的"
  但它实际上不会被再次访问 → 它"挡"在淘汰队列的前面
  → 真正热的 key 可能被淘汰

LFU 的优势：
  LFU 不只看"最近有没有访问"，而看"历史访问频率"
  偶尔被访问一次的 key → 频率很低 → 即使最近访问过，也会被淘汰
```

### 5.5 Redis 的 LRU 实现——为什么不是"精确 LRU"

**Redis 没有实现精确的 LRU（不像 LeetCode 146），而是近似的。**

```
精确 LRU 的做法：
  ┌──────────────────────┐
  │ [最近] [次近] ... [最久] │  ← 双向链表，每次访问移到头部
  └──────────────────────┘
  淘汰时删尾部 = 最久没用的
  
  问题：需要维护全局双向链表 → 每次访问都要移动节点 → 额外内存 + CPU 开销
        Redis 可能有几百万个 key → 链表操作量巨大


Redis 的近似 LRU 做法：
  ① 随机抽取 N 个 key（N 默认 = 5，可配置 maxmemory-samples）
  ② 在这 N 个 key 中，找出"最久没被访问的"那个（idle 时间最长）
  ③ 淘汰它

  优点：不用维护全局链表，O(N) 变成 O(5)（N 很小）
  缺点：不是 100% 精確——可能淘汰的不是全局最久没用的那个

  精度：N=5 时约 85% 命中理论最优，N=10 时约 98%
        增大 N → 更接近精确 LRU → 但 CPU 增加
```

**Redis 怎么知道一个 key "多久没被访问了"？**:rofl::rofl::rofl::rofl::rofl:

```
答案：redisObject 的 lru 字段（24 bit）

Redis 中每个 key 的 value 都是一个 redisObject：
  ┌──────────┬──────────┬────────────────────┬──────────────┐
  │ type     │ encoding │ lru (24 bit)        │ ptr          │
  │ 4 bit    │ 4 bit    │ 当前时间戳的低 24 位 │ 指向实际数据  │
  └──────────┴──────────┴────────────────────┴──────────────┘

  每次 key 被访问时 → lru 字段被更新为当前时间的低 24 位
  判断"多久没被访问" = (当前时间 - lru 记录时间) 的差值
  差值越大 → 越久没被访问 → 优先淘汰
```

### 5.6 LRU 的近似淘汰流程图

```
内存满了，需要淘汰一个 key：

  ① 从所有 key（allkeys-lru）或有 TTL 的 key（volatile-lru）中
     随机抽取 N 个 key（N = maxmemory-samples，默认 5）

  ② 对每个抽取的 key：
     读 redisObject.lru 字段 → 得到上次访问时间
     计算 idle = 当前时间 - lru时间
     找 idle 最大的 key

  ③ 淘汰 idle 最大的那个 key

  ④ 如果淘汰后内存还是不够 → 重复①②③
     直到内存降到 maxmemory 以下或淘汰到最小限制
```

### 5.7 LFU 在 Redis 中的实现——复用了 lru 字段

Redis 的 LFU 用了一个巧妙的方法——把本来用于 LRU 的 24-bit lru 字段拆成两部分来用于 LFU：

```
24-bit lru 字段在 LFU 模式下的拆分：

  ┌──────────────────────┬────────────────────┐
  │ 上次衰减时间（16 bit） │ 访问频率计数器（8 bit）│
  │ Unix 分钟时间戳的低16位│ 0~255 的访问计数     │
  └──────────────────────┴────────────────────┘

访问频率计数器（8 bit = 0~255）的更新逻辑：

  ① key 被访问 → 计数器 +1
     但并不是每次访问都 +1——是概率增加（counter 越大，增加概率越低）
     这防止频繁访问的 key 很快就冲到 255 封顶

  ② 计数器随时间衰减：
     每过一定时间（由 lfu-decay-time 控制），计数器 -1
     这是 LFU 的精髓——"不经常使用"就是要惩罚"当前频率低"
     之前频率高但最近不怎么访问的 → 计数器持续衰减 → 最终被淘汰
```

### 5.8 淘汰策略选型建议:rofl::rofl:

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| **纯缓存，数据能丢** | `allkeys-lru` | 自动淘汰最近最少用的，最省心 |
| **缓存 + 热点不均匀** | `allkeys-lfu` | 避免"偶访一次就赖着不走"的问题 |
| **缓存 + 有永久数据** | `volatile-lru` | 没设 TTL 的永久数据不会被淘汰 |
| **不知道选什么** | `allkeys-lru` | 最安全的默认选择 |

### 5.9 相关配置

```bash
# redis.conf

# ① 最大内存（必须设置！默认 0 = 不限制 = 撑爆物理内存）
maxmemory 2gb

# ② 淘汰策略
maxmemory-policy allkeys-lru

# ③ LRU/LFU 采样精度（越大越精确，但 CPU 越高）
maxmemory-samples 5           # 默认 5，建议不超过 10

# ④ LFU 相关（只有选 LFU 策略才生效）
lfu-log-factor 10             # 计数器增长的对数因子（越大增长越慢）
lfu-decay-time 1              # 计数器衰减周期（1 = 每分钟衰减 1）
```

## 第六章：主从模式下过期键的处理

### 6.1 问题：Master 的 key 过期了，Slave 怎么同步？:rofl::rofl::rofl:

主从架构下，过期 key 的处理是一个需要特殊设计的点——因为 Master 和 Slave 都有同一份数据的副本。

```
设计的核心原则：由 Master 统一控制过期，Slave 被动服从

这避免了两种服务器都有自己的"时钟判断"导致的不一致——
Master 认为 key 过期了，Slave 认为还没过期，就会产生歧义。
```

### 6.2 Master 如何处理过期 key

```
Master 上的过期流程：

  ① Master 上某个 key 过期了（被惰性删除或定期删除检测到）
  ② Master 正常删除该 key（和单机 Redis 一样）
  ③ Master 在执行删除时，会在 AOF 中追加一条 DEL key 命令
  ④ 这个 DEL 命令通过主从复制同步给 Slave

奴隶最终一致：
  Slave 收到 DEL 命令 → 删除本地的该 key 副本
  → Slave 不自己判断"key是否过期"，完全由 Master 的 DEL 命令控制
```

### 6.3 Slave 如何处理读请求（过期保护）:rofl::rofl::rofl::rofl::rofl:

```
关键问题：如果 Slave 收到客户端的读请求（READ 命令），
         而 key 在 Master 上已经过期但 Slave 还没收到 DEL 命令，怎么办？

Slave 的处理方式（Redis 3.2+）：

  ┌──────────────────────────────────────────────┐
  │ Slave 收到 GET expired_key                   │
  │                                              │
  │ ① 检查 expires dict → key 有 TTL             │
  │ ② 检查过期时间 → 已过期！                     │
  │                                              │
  │ Slave 的处理（和 Master 不同！）：              │
  │   Slave 不主动删除 key                        │
  │   而是返回 null 给客户端（表现得像 key 不存在）  │
  │                                              │
  │ 为什么不直接删？                               │
  │   如果 Slave 直接删 → Master 还没发 DEL       │
  │   → Master 和 Slave 不一致                    │
  │   → Slave 自作主张，违背了"Master 统一控制"原则│
  └──────────────────────────────────────────────┘

  这样设计的好处：
    - Slave 对客户端提供一致的数据视图（读到过期 key 返回 null）
    - Slave 不主动修改自己的数据（等 Master 的 DEL 命令才真正删除）
    - 无论 Master 的 DEL 是否到达，Slave 不会错误地提供已过期数据
```

### 6.4 完整的主从过期 key 处理流程

```
═══════════════════════════════════════════════════════════════
        主从架构下过期键处理的完整时间线
═══════════════════════════════════════════════════════════════

              Master                                    Slave
              ──────                                    ─────

  t=0     key=temp 设了 TTL=300s                    key=temp 也设了 TTL=300s
          (主从复制同步)                              (同步过来的)

  t=300s  惰性删除检测：temp 过期了
          ├─ DEL temp（删除本地的 temp）
          ├─ AOF 追加：DEL temp
          └─ 将 DEL temp 同步给 Slave  ────────────▶ 收到 DEL temp
                                                     └─ 删除本地的 temp
```

## ⭐️ 面试题汇总

**Q1: Redis 的过期 key 是怎么被删除的？**

> 三种策略配合：① 惰性删除——key 被访问时才检查是否过期，省 CPU 但可能残留"死 key"；② 定期删除——每 100ms 随机抽 20 个 key 检查，过期比例 > 25% 则循环，每次最多 25ms；③ 内存淘汰——内存满了触发 maxmemory-policy 淘汰。惰性 + 定期是正常机制，内存淘汰是兜底。

**Q2: 为什么定期删除是抽 20 个而不是遍历所有？**

> 如果 expires dict 有 100 万 key → 遍历所有 O(N) 可能要几秒 → 主线程被卡住。随机抽 20 个 O(20) 微秒级，如果过期比例 > 25% 说明"过期确实很严重"，再抽一轮继续清。限时 25ms 保证不管过期多严重，都不会卡主线程超过 25ms。

**Q3: 大量 key 集中过期导致 Redis 延迟抖动，怎么排查和解决？**

> 排查：`INFO stats` 看 `expired_keys` 是否在短时间内猛增。解决：TTL 加随机偏移（最直接）+ 程序分批加载不一次性全量 + 如果已发生→等它清完，或手动 SCAN + DEL 部分过期 key。

**Q4: Redis 的 LRU 和 LeetCode 上的 LRU 有什么区别？**

> LeetCode LRU 是精确 LRU（双向链表 + HashMap），需要维护全局有序结构，O(1) 但内存和 CPU 开销大。Redis 是近似 LRU——随机抽 N 个 key，淘汰其中 idle 时间最长的。N=5 时约 85% 命中理论最优，N=10 时约 98%。内存开销几乎为零（复用 redisObject 的 lru 字段）。

**Q5: allkeys-lru 和 volatile-lru 的区别？**

> allkeys-lru：从**所有 key** 中淘汰。volatile-lru：从**设置了 TTL 的 key** 中淘汰。如果所有 key 都设了 TTL → 两者等价。如果有部分 key 没设 TTL（当作永久数据）→ allkeys 可能淘汰它，volatile 不会。

**Q6: LFU 比 LRU 好在哪里？**

> LRU 只看"最近有没有访问"→ 偶尔被访问一次的冷 key 可能排在前面导致热 key 被淘汰。LFU 看"历史访问频率"→ 频繁访问的 key 频率高，不会被淘汰；偶尔访问的 key 频率低，即使最近刚访问过也会随时间衰减并被淘汰。

**Q7: noeviction 策略下，Redis 内存满了会怎样？**

> 读请求正常（GET 不占额外内存）。写请求（SET/INCR/LPUSH 等所有会占用更多内存的命令）直接报错 `OOM command not allowed when used memory > 'maxmemory'`。DEL 等释放内存的命令可以正常执行。
