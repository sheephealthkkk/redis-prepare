## 第〇章：前置知识——什么是持久化？

### 0.1 Redis 的数据在哪？

```
Java 程序员的直觉：
  HashMap<String, Object> cache = new HashMap<>();
  cache.put("user:1", user);
  // 程序退出 → HashMap 没了 → 数据没了

Redis 是一样的问题：
  所有数据在内存中 → 进程崩溃 / 服务器断电 → 一切消失
```

### 0.2 持久化 = 把内存数据写到磁盘

```
两种完全不同的思路：

  RDB（快照）：每隔一段时间，咔嚓一下把整个内存"拍照"存到磁盘
              → dump.rdb（一个二进制压缩文件）

  AOF（日志）：每一条修改了数据的命令，都追加到日志文件末尾
              → appendonly.aof（一个文本文件）
```

## 第一章：fork()——理解 Redis 持久化的钥匙

### 1.1 先理解：什么是"进程"？

**Java 程序员对进程的认知：** 你启动一个 `java -jar myapp.jar` → 操作系统创建一个进程 → JVM 在这个进程里运行。

```java
// Java 里你几乎感知不到"进程"的存在，因为 JVM 帮你隔离了一切
// 你在 Java 里操作的是"对象"——new User()、map.put()，这些都是 JVM 内部的事
// 但 Redis 是用 C 写的，C 是可以直接操作进程的
```

**进程 = 一个正在运行的程序实例，拥有独立的内存空间。**

### 1.2 虚拟内存——每个进程都有一张"假地图"

这是理解 fork 最关键的前置知识。操作系统给每个进程一个"幻觉"——"你拥有全部内存"。

```
物理世界（真实的 RAM 芯片）：
  ┌──────────────────────────┐
  │ RAM: 16GB                │
  │ 实际地址: 0x0000 ~ 0x3FFF │
  │ 这里塞了所有进程的数据       │
  └──────────────────────────┘

进程眼中的世界（虚拟内存）：
  ┌──────────────────────────┐
  │ "我有 128TB 的空间！"    │
  │ 我的地址 0x1000 = 那块物理内存 0x2000（操作系统做的翻译）
  │ 每个进程有自己的虚拟地址空间
  └──────────────────────────┘
```

**为什么需要虚拟内存？** 两个原因：① 隔离——进程 A 不能偷看进程 B 的内存；② 简化——每个程序以为自己独占所有内存，不用关心"别人在哪"。

### 1.3 页表——虚拟地址到物理地址的"翻译字典"

虚拟地址 → 物理地址的翻译，靠的是一张叫**页表（Page Table）**的表

```
页表 = HashMap<虚拟页号, 物理页号>

举个具体的例子：

  进程 A 的页表：
    虚拟页 0  → 物理页 100   ← 当进程 A 读写地址 0~4095 时，实际读写的是物理地址 100
    虚拟页 1  → 物理页 200
    虚拟页 2  → 物理页 50
    ...

  物理内存按"页"（Page）切割，每页固定 4KB
  一个 10GB 的 Redis 实例 = 10GB / 4KB = 262 万页
  每页需要一个页表条目（8 字节）= 262 万 × 8 字节 ≈ 21MB 的页表
```

**这是理解 fork 开销的核心：fork 拷贝的正是这张"页表"。**

### 1.4 fork() 到底做了什么？

**fork() = 创建一个和父进程一模一样的子进程。** Java 无法直接做到这一点（JVM 没有 fork），所以在 Java 世界里没有对应物。

```
fork() 的执行过程（一步一步）：

  fork() 调用之前：
    ┌──────────────┐
    │ 父进程(Redis) │
    │ 内存: 10GB   │
    │ 页表: ~21MB  │  ← 虚拟→物理的翻译字典
    │ 代码/数据... │
    └──────────────┘

  fork() 执行中：
    ① 操作系统创建一个新的"进程控制块"(PCB)——新进程的身份信息
    ② 操作系统拷贝父进程的「页表」给子进程
       （每个页表条目 8 字节，262 万条 = 约 21MB 的数据拷贝）
    ③ fork() 返回到父进程：返回子进程的 PID
       fork() 返回到子进程：返回 0

  fork() 完成后：
    ┌──────────────┐     ┌──────────────────┐
    │ 父进程(Redis) │     │ 子进程(BGSAVE)   │
    │ 内存: 10GB   │     │ 页表: 指向相同的  │
    │ 页表: ~21MB  │     │ 物理页！          │
    │ 继续处理请求  │     │ 开始写 dump.rdb   │
    └──────────────┘     └──────────────────┘
        共享物理内存的 10GB！（还没有复制，只是共享）
```

**关键理解——三个"不等于"：**

> fork **不等于**拷贝整个 10GB 内存。（只拷贝了 ~21MB 页表）  
> fork **不等于**瞬间完成。（拷贝 21MB 页表需要几十到几百毫秒）  
> fork **不等于**父子进程独立占有内存。（通过 COW 共享——马上讲到）

### 1.5 为什么 fork() 极快但又不是瞬间？

```
fork() 所做的工作：
  ① 创建新进程控制块 → 可忽略（微秒级）
  ② 拷贝父进程页表 → 这就是主要开销！

拷贝页表的时间：
  页表大小 = Redis 内存大小 / 4KB × 8 字节
  10GB → 262 万页 × 8B = 21MB 页表 → 拷贝 ~50ms
  20GB → 524 万页 × 8B = 42MB 页表 → 拷贝 ~120ms
  50GB → 1310 万页 × 8B = 105MB 页表 → 拷贝 ~300ms

这就是为什么生产环境建议单 Redis 实例不超过 10-16GB 的原因之一：
内存越大 → 页表越大 → fork 阻塞主线程越长
```

**fork 期间发生了什么？**:o::o::o::o::o::o::o::o::o::o::o: Redis 主线程在调用 `fork()` 时被操作系统挂起，等待 OS 拷贝完所有页表后才恢复执行。这期间的几十到几百毫秒，Redis 无法处理任何客户端请求。

### 1.6 Redis 中哪些操作会触发 fork()？:o::o::o::o::o:

```
触发 fork() 的三种情况：

  ① BGSAVE（RDB 持久化）
     触发条件: save 配置满足 / 手动 BGSAVE 命令 / 主从全量同步
     子进程任务: 把当前内存数据写入 dump.rdb
     主进程任务: 继续处理客户端请求

  ② BGREWRITEAOF（AOF 重写）
     触发条件: AOF 文件大小超过阈值自动触发 / 手动 BGREWRITEAOF 命令
     子进程任务: 扫描当前内存数据，生成等价的最小化 AOF 命令
     主进程任务: 继续处理客户端请求，写命令记录在 AOF 重写缓冲区

  ③ 主从全量同步
     触发条件: Slave 首次连接 Master / 复制偏移量差距过大
     子进程任务: 生成 RDB 快照发给 Slave
     主进程任务: 继续处理客户端请求
```

## 第二章：Copy-on-Write（COW）——父子进程如何共享内存

### 2.1 问题：fork 之后怎么办？

fork 之后，父子进程**共享同一块物理内存**。但父进程（Redis 主线程）还在处理大量请求——不停修改内存数据。子进程在写 dump.rdb——需要稳定的数据快照。这不就冲突了吗？

```
矛盾：

  父进程：我要修改 key "counter" 的值（从 5000 改成 5001）
  子进程：我在把"counter"写到 dump.rdb，我需要稳定的 5000

  如果父进程直接改物理内存 → 子进程会读到"改到一半的脏数据" → 写出来的 RDB 不可靠
```

### 2.2 COW 的解决思路

**Copy-on-Write = "谁要写，谁先复制那一页，在复制后的页上改，别人继续看原页"。**:o::o::o::o::o::o::o::o::o::o::o::o::o::o::o:

```
生活类比：
  父子两人共享一本书。
  规则是：谁要在某一页上写笔记，谁就先把那一页"复印"一份，在自己的复印件上写。
  不想写的那个人继续看原书。

COW 的完整流程（用上面的例子）：

  fork 刚完成时：
    父进程 ──┐          物理内存：
             ├──────▶  [页0][页1][页2][页3] ... （共享中）
    子进程 ──┘

  父进程要修改"counter"（位于页2上）：
    ① CPU 检测到"父进程要写一个被子进程共享的页"
       （通过页表的"写保护位"）
    ② CPU 触发"页故障（Page Fault）"中断
    ③ 操作系统内核接管：
       a. 分配一页新的物理内存（页2'）
       b. 把"页2"的内容复制到"页2'"
       c. 更新父进程的页表：虚拟页2 → 新的物理页2'
       d. 取消新页2'的写保护，允许父进程写入
    ④ 父进程继续执行，在页2'上修改 counter → 5001
    ⑤ 子进程仍然查看原来的"页2"（counter = 5000）

  COW 完成后：
    父进程 ──▶ [页0][页1][页2'][页3] ...  ← 页2' 是新分配的
                        ↑
    子进程 ──▶ [页0][页1][页2][页3] ...  ← 子进程仍指向原页2
```

### 2.3 COW 的内存开销——关键计算:o::o::o::o::o::o::o::o::o::o::o::o::o::o::o::o::o:

```
正常情况（子进程写盘期间，父进程不怎么改数据）：
  父进程的写操作很少 → COW 触发的页面很少
  → 额外内存消耗 ≈ 0（只有被改动的页才复制）

最坏情况（子进程写盘期间，父进程修改了所有数据）：
  所有 10GB 的页面都会被 COW 复制
  → 额外内存消耗 ≈ 10GB
  → 总物理内存 = 父的 10GB + COW 产生的新 10GB = 20GB！

实际生产：
  BGSAVE 通常持续几秒到几十秒（取决于内存大小和磁盘速度）
  只有写操作改到的页才会触发 COW
  对于读多写少的系统 → 通常额外消耗 10%~20%
  对于写密集型系统 → 可能达到 50%~80%
```

### 2.4 COW 灾难场景

```
场景：
  Redis 实例: 占用 16GB 内存
  服务器物理内存: 32GB
  业务特点: 每秒 5000 次 SET 操作（写密集型）

灾难时间线：
  ① 0 秒: BGSAVE 触发 → fork 子进程（此时父子共享 16GB）
  ② 1 秒: 5000 次 SET → ~200MB 的数据被修改 → COW 分配 ~200MB
  ③ 5 秒: 累计 ~1GB 的 COW 新增内存
  ④ 10 秒: 累计 ~2GB
  ⑤ 15 秒: 累计 ~3GB
  ⑥ 20 秒: BGSAVE 完成，子进程退出，COW 新分配的 ~4GB 自动释放

  峰值内存 = Redis 的 16GB + COW 的 ~4GB = ~20GB
  离 32GB 的物理内存还有距离 → 安全


但如果 Redis 占用了 24GB（在 32GB 物理机上）：
  ① 0 秒: fork → 父子共享 24GB
  ② 10 秒: COW 产生 ~3GB → 物理内存峰值 = 24 + 3 = 27GB
  ③ 15 秒: COW 产生 ~5GB → 物理内存峰值 = 24 + 5 = 29GB
  ④ 18 秒: COW 产生 ~8GB → 物理内存峰值 = 24 + 8 = 32GB ⚠️ OOM！
  → 操作系统 OOM Killer 杀死 Redis → 生产事故！
```

**核心教训：物理内存必须 ≥ maxmemory × 1.5（给 COW 留出空间）。**:o::o::o::o::o::o:

## 第三章：THP（透明大页）——把 COW 的问题放大 512 倍

### 3.1 什么是内存"页"？

操作系统管理内存的最小单位是**页（Page）**。默认一页 = 4KB:rocket:。

```
为什么需要"页"？
  物理内存 16GB → 如果按字节管理 → 160 亿个单元 → 管理开销太大
  按页管理 → 16GB / 4KB = 419 万页 → 管理成本可接受

COW 的复制粒度 = 页：
  父进程修改了 byte[0]（在"页A"上）
  → 整个"页A"（4KB）被复制

这个 4KB 就是 COW 的最小开销
```

### 3.2 THP 是什么？:rocket:

**THP（Transparent Huge Pages）= 操作系统把 512 个连续的 4KB 页合并成 1 个 2MB 的"大页"。**

```
标准页（4KB）：
  ┌───┐┌───┐┌───┐┌───┐... (512 个独立的 4KB 页)
  │ 0 ││ 1 ││ 2 ││ 3 │
  └───┘└───┘└───┘└───┘
  页表条目: 512 条（每条 8 字节 = 4096 字节）

THP 页（2MB）：
  ┌─────────────────────────┐
  │   一个 2MB 的大页       │
  │   (包含 512 个 4KB 页)  │
  └─────────────────────────┘
  页表条目: 1 条（8 字节）
  
  THP 的优势：减少页表条目 → 减少 TLB miss → CPU 更快
  THP 的初衷：为内存密集型应用（如数据库）提升性能
```

### 3.3 THP × COW = 内存雪崩

**这是 Redis 在生产环境中最隐蔽的坑。**

```
没有 THP（页 = 4KB）：
  Redis 修改了 1 字节数据 → 复制该 4KB 页 → COW 开销 = 4KB

有 THP（页 = 2MB）：
  Redis 修改了 1 字节数据 → 复制该 2MB 大页 → COW 开销 = 2MB
  2MB / 4KB = 512 倍！

具体场景：
  BGSAVE 子进程正在写盘，主进程每秒 5000 次 SET
  每次 SET 可能改到不同的内存页

  没有 THP: 5000 次 SET → 最多 5000 × 4KB = 20MB 的 COW 开销
  有 THP:   5000 次 SET → 可能 5000 × 2MB = 10GB 的 COW 开销！

  10GB 的 COW → 物理内存瞬间暴增 10GB → 本来够用的 32GB 物理内存不够了
  → OOM Killer 杀死 Redis → 生产事故！
```

### 3.4 解决方案

```bash
# ① 检查当前 THP 状态
cat /sys/kernel/mm/transparent_hugepage/enabled
# 输出: [always] madvise never
#       ↑ always = THP 已开启

# ② 临时关闭（立即生效）
echo never > /sys/kernel/mm/transparent_hugepage/enabled

# ③ 永久关闭（写入系统启动脚本）
# /etc/rc.local 中添加：
echo never > /sys/kernel/mm/transparent_hugepage/enabled

# ④ Redis 启动时也主动尝试关闭（Redis 6.x+ 会打印警告日志）
# 但最好在操作系统层面关闭，不要依赖 Redis
```

**面试必记一句话：** "THP 会让 COW 的复制粒度从 4KB 膨胀到 2MB——放大 512 倍——BGSAVE 或 AOF 重写期间的大量写入可能导致内存瞬间耗尽。":rocket::rocket::rocket::rocket::rocket::rocket::o::o:

## 第四章：RDB 持久化

### 4.1 RDB 做了什么？

**就是上面讲的 fork + COW 的完整应用。** 在某个时刻，创建一个子进程，子进程负责遍历所有内存数据，写入 .rdb 二进制压缩文件。

```
Redis 内存状态（某一时刻）→ 子进程写入 dump.rdb：
  key: "user:1"  → {name:"张三", age:25}
  key: "counter" → 5000
  key: "rank"    → [alice:100, bob:95]
  ...
         │
         │ fork → 子进程遍历所有数据
         ▼
  ┌───────────┐
  │ dump.rdb  │  二进制压缩文件（LZF 压缩）
  │  ~100MB   │
  └───────────┘
```

### 4.2 SAVE（阻塞）vs BGSAVE（fork 非阻塞）:rocket::rocket::rocket::rocket::rocket::rocket::rocket::rocket::o:

```
SAVE 命令：
  ┌───────────────────────────────────────┐
  │ Redis 主线程                          │
  │ 开始 SAVE ───────────────────▶        │
  │ 遍历所有 key → 写入 dump.rdb        	  │
  │ 在这个过程中：任何客户端请求都排队！    	 │
  │ 10GB 数据 = 可能要几秒到几十秒        	 	│
  │ 生产环境绝对不能用 SAVE！              	│
  └───────────────────────────────────────┘

BGSAVE 命令：
  ┌──────────────────┐    ┌──────────────────┐
  │ 主线程            │    │ 子进程            │
  │                  │    │                  │
  │ ① fork() ───────┼───▶│ ② 遍历内存数据   │
  │ (阻塞 ~100ms)    │    │   写入 dump.rdb   │
  │                  │    │                  │
  │ ③ fork 返回后     │    │   写盘过程中：    │
  │   继续处理请求    │    │   父进程修改数据 → │
  │   SET/GET/INCR.. │    │   COW 复制保护    │
  │                  │    │                  │
  └──────────────────┘    └──────────────────┘
```

### 4.3 RDB 触发条件:o::o::o::o::o:

```bash
# redis.conf 中的默认配置

# ① 自动触发（满足任意就 BGSAVE）
save 900 1       # 15 分钟内 ≥ 1 次修改
save 300 10      # 5 分钟内 ≥ 10 次修改
save 60 10000    # 1 分钟内 ≥ 10000 次修改

# ② 手动触发
#    BGSAVE（推荐）
#    SAVE  （禁止生产使用）

# ③ 主从全量同步
#    Slave 连接 Master → Master 执行 BGSAVE
```

### 4.4 RDB 配置详解:rocket::rocket::rocket::rocket:

```bash
# ===== redis.conf RDB 相关 =====

# ① 自动 BGSAVE 触发条件
save 900 1
save 300 10
save 60 10000

# ② 如果 BGSAVE 失败，拒绝所有写入
stop-writes-on-bgsave-error yes
# 磁盘满了 → BGSAVE 失败 → 拒绝所有写操作
# 防止"磁盘坏了但一直没人知道"的情况

# ③ LZF 压缩 RDB 文件
rdbcompression yes
# 内存数据的字符串部分用 LZF 算法压缩
# CPU 消耗很小（微秒级），文件大小减少 ~30-50%

# ④ CRC64 校验
rdbchecksum yes
# RDB 文件末尾存 CRC64 校验和，加载时验证文件完整性

# ⑤ 文件名和路径
dbfilename dump.rdb
dir /var/lib/redis
```

### 4.5 RDB 的优缺点

| 优点 | 缺点 |
|------|------|
| 文件紧凑（二进制压缩） | 两次 RDB 之间的数据可能丢失（几分钟到十几分钟）:rocket:**在两次快照之间，所有写操作都只存在于内存中。** 如果还没等到下一次快照，服务器就意外断电或崩溃了，内存里的最新数据自然就消失了。 |
| 恢复速度快（直接加载二进制） | fork 阻塞（页表拷贝）:rocket: |
| 适合备份/灾备/主从同步 | COW 内存开销:rocket: |
| 对性能影响可控（子进程写盘） | 不适合实时持久化 |

## 第五章：AOF 持久化

### 5.1 AOF 做了什么？

**AOF = 把每条"修改了数据"的写命令追加到文本文件中。** 重启时逐条重放命令恢复数据。

```
Redis 执行命令的过程：

  客户端: SET user:1 "张三"
         │
         ▼
    Redis 主线程:
    ① 执行 SET → 内存中 user:1 的值变成 "张三"
    ② 把这条命令序列化成 RESP 格式 → 追加到 AOF 缓冲区
         │
         ▼ (根据 appendfsync 配置决定何时写磁盘)
    appendonly.aof:
    ┌──────────────────────────┐
    │ SET counter 5000         │
    │ INCR counter             │
    │ SET user:1 张三          │
    │ LPUSH queue msg1         │
    │ ...                      │
    └──────────────────────────┘
```

### 5.2 AOF 工作流（四个步骤）

```
AOF 完整数据流：

  客户端写命令
       │
       ▼
  [步骤 1: 执行命令] → 修改内存中的数据
       │
       ▼
  [步骤 2: 追加到 AOF 缓冲区] → aof_buf（内存中的一块区域）
       │                        类似 Java 的 BufferedWriter
       ▼
  [步骤 3: 缓冲区 → 磁盘] → 这一步由 appendfsync 控制
       │                   always / everysec / no
       ▼
  [步骤 4: AOF 重写] → AOF 文件越来越大 → 定期压缩
```

### 5.3 三种 fsync 策略

这是决定"数据安全性 vs 性能"的核心配置：

```bash
# redis.conf
appendfsync everysec  # 推荐
```

```
┌──────────────┬──────────────┬──────────────┬──────────────┐
│              │   always     │  everysec    │    no        │
├──────────────┼──────────────┼──────────────┼──────────────┤
│ 何时写盘     │ 每条命令后   │ 每 1 秒一次   │ OS 决定      │
│             │              │             │ (通常30s)    │
│ 性能         │ 最慢（几千QPS）│ 几乎无损     │ 最快         │
│ 丢失风险     │ 几乎 0       │ ≤ 1 秒数据   │ ≤ 30 秒数据  │
│ 磁盘 I/O     │ 极高（每条一次）│ 低（每秒一次）│ 最低         │
│ 适用场景     │ 金库/支付     │ 99% 的生产    │ 纯缓存       │
└──────────────┴──────────────┴──────────────┴──────────────┘
```

**为什么 everysec 是"最多丢 1 秒"？**:rocket::rocket::rocket::o::o::o::o::o::o::o::o:

```
AOF 刷盘的实现：不是主线程直接 fsync——那样会阻塞！
  而是 Redis 后台的 BIO-2 线程每秒执行一次 fsync

  主线程：把命令追加到 aof_buf → 立即返回
  后台线程：每秒把 aof_buf 写入 AOF 文件 → 调用 fsync

  如果恰好在这一秒间隔内断电 → aof_buf 中的内容丢失 → 最多 1 秒
```

和mysql的redolog一毛一样，从log buffer到页缓存，页缓存到磁盘日志文件才是真正持久化，而且也是，一条一次，一秒一次，或者os控制写:o::o::o::o::o::o::o:**MySQL 的 redo log 和 Redis 的 AOF，在写入路径上确实是“神似形不似”**。它们都用了缓存层，都有类似的三个参数来控制刷盘时机，

| 对比维度            | MySQL redo log (`innodb_flush_log_at_trx_commit`)            | Redis AOF (`appendfsync`)                                    |
| :------------------ | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **参数 1 (最安全)** | **设置为 1** 每次事务提交都 `write` + `fsync`。 绝对不丢已提交事务的数据。 | **设置为 always** 每个写命令都 `write` + `fsync`。 绝对不丢已执行的命令。 |
| **参数 2 (折中)**   | **设置为 2** 每次事务提交只 `write`，每秒 `fsync` 一次。 MySQL 进程崩溃不丢数据，但**操作系统崩溃会丢 1 秒**的数据。 | **设置为 everysec** 每秒 `write` + `fsync` 一次。 默认配置。**宕机会丢 1 秒**的命令。 |
| **参数 3 (最危险)** | **设置为 0** 事务提交时**连 `write` 都不做**，每秒由后台线程 `write` + `fsync`。 MySQL 进程崩溃会丢失最近 1 秒的事务。 | **设置为 no** 只 `write`，不主动 `fsync`，完全交给操作系统刷盘。 数据安全性依赖操作系统，**极易丢数据**。 |

### 5.4 AOF 重写——也用 fork()！:rocket::rocket::o::o::o::o::o::o::o::o::o::o::o::o::o::o::o::o:

**这是 AOF 和 RDB 的联动点——都用同一个 fork 机制。**

```
为什么要重写？
  AOF 记录的是"所有历史写命令"——运行一个月 → AOF 几百 GB
  很多中间过程是冗余的：
    INCR counter (× 5000 次) → 最终 counter = 5000
    重写后只需一行："SET counter 5000"

AOF 重写的执行流程（用 fork）：

  ┌───────────────────────┐     ┌───────────────────────────┐
  │ 主线程                │     │ 子进程（fork出来的）       │
  │                       │fork │                           │
  │ ① fork() ────────────┼────▶│ ② 扫描当前内存中的最终数据  │
  │                       │     │   生成最精简的 SET 命令     │
  │ ③ 子进程写盘期间，     │     │   写入新的 AOF 文件        │
  │   新来的写命令除了     │     │                           │
  │   正常执行，还写入     │     │                           │
  │   "AOF 重写缓冲区"     │     │                           │
  │                       │     │                           │
  │ ④ 子进程写完 → 通知   │◀────│                           │
  │   主进程把 AOF 重写缓  │     │                           │
  │   冲区的命令追加到     │     │                           │
  │   新 AOF 文件末尾      │     │                           │
  │                       │     │                           │
  │ ⑤ 原子 rename →      │     │                           │
  │   新 AOF 替换旧 AOF    │     │                           │
  └───────────────────────┘     └───────────────────────────┘
  
  同样发生 fork 阻塞！
  同样发生 COW！
  同样受 THP 影响！
```

**触发 AOF 重写的条件：**

```bash
auto-aof-rewrite-percentage 100    # AOF 文件比上次重写后增长 100%（翻倍）
auto-aof-rewrite-min-size 64mb     # 且 AOF 文件 ≥ 64MB（太小不触发）
```

### 5.5 为什么 AOF 在执行命令之后记录？:rocket::rocket::rocket::rocket::rocket:

wal:说的是在执行前记录，先记录，如果事物不满足再删除:o::o::o::o::o::o::o::o::o::o::o::o::o::o::o::o:

```
Redis AOF 的顺序：
  ① 执行命令 → ② 执行成功 → ③ 记录到 AOF

为什么不是先记录再执行（像 MySQL WAL 那样）？三个原因：

  1. 没有任何必要——MySQL 需要 WAL 是因为"事务写到一半崩溃"要能恢复
     Redis 单条命令直接执行 = 原子（单线程执行），不需要"恢复未完成的事务"

  2. 语法检查已在执行阶段完成——执行成功 = 命令合法，不需要在 AOF 中再验证

  3. 执行失败的命令不记录——如果命令报错了，不应该在 AOF 里留日志
     （先写日志的话，你不知道这条命令后来会不会报错）
```

### 5.6 AOF 文件完整性验证

**纯 AOF 文件：** 没有校验和。Redis 加载时用状态机逐字节解析 RESP——发现格式不对就丢弃损坏部分及后续所有内容。

**混合持久化文件（Redis 4.0+）：** 文件 = RDB 快照 + 增量 AOF。Redis 7.0+ 支持多段 CRC64 校验——每段有独立校验和，某段损坏只丢该段。

```bash
# 手动修复损坏的 AOF 文件
redis-check-aof --fix appendonly.aof
# 丢弃所有在第一个损坏段之后的内容
```

### 5.7 AOF 完整配置

```bash
# ===== redis.conf =====

appendonly yes                                    # 开启 AOF
appendfilename "appendonly.aof"                   # AOF 文件名
appendfsync everysec                              # fsync 策略
no-appendfsync-on-rewrite yes                     # BGSAVE/BGREWRITEAOF 时暂停 fsync
auto-aof-rewrite-percentage 100                   # 文件翻倍时触发重写
auto-aof-rewrite-min-size 64mb                    # 至少 64MB 才重写
aof-use-rdb-preamble yes                          # 混合持久化
aof-load-truncated yes                            # 文件尾部截断仍可加载
```

## 第六章：RDB vs AOF vs 混合持久化

| 维度 | RDB | AOF | 混合持久化 |
|------|-----|-----|-----------|
| **文件格式** | 二进制压缩 | RESP 文本 | RDB 二进制 + AOF 文本 |
| **文件大小** | 小（压缩）:o: | 大（不压缩时）:o: | 中（RDB 部分压缩） |
| **恢复速度** | 快（直接载入）:o::o: | 慢（逐条回放命令）:o::o::o: | 较快（RDB 部分 + 少量 AOF） |
| **数据安全** | 两次快照间丢失 | ≤ 1 秒（everysec） | ≤ 1 秒 |
| **fork 阻塞** | 每次 BGSAVE | 仅重写时 | 仅重写时 |
| **COW 风险** | 有 | 仅重写时有 | 仅重写时有 |
| **适用** | 备份、主从同步:o::o: | 不容忍丢数据 | **生产推荐** |

### 6.1 为什么会有混合持久化？——纯 RDB 和纯 AOF 各自的问题

混合持久化（Redis 4.0+）的诞生，是因为纯 RDB 和纯 AOF 各有一个要命的缺陷：

```
纯 RDB 的问题——数据丢失窗口大：

  假设 save 配置是 "save 900 1"（15 分钟内有 1 次修改就 BGSAVE）
  
  时间线：
    12:00 → BGSAVE 完成，dump.rdb 包含截止 12:00 的所有数据
    12:01 → SET user:new  ← 新数据进入内存
    12:05 → SET order:new ← 新数据进入内存
    12:14 → Redis 宕机！
    
  恢复后：dump.rdb 是 12:00 的快照 → user:new 和 order:new 都丢了
  → 丢了 14 分钟的数据！

  纯 RDB 不适合对数据丢失敏感的场景


纯 AOF 的问题——恢复太慢：

  运行一个月后，AOF 文件记录了几十亿条命令。
  如果 AOF 重写还没来得及跑，或者重写频率不够。
  重启加载 = 逐条执行几十亿条命令。
  
  1000 万条命令 → 约需 5-10 分钟恢复！
  
  纯 AOF 的恢复速度跟不上业务需求


混合持久化的解法：
  ① 使用 BGREWRITEAOF（同样 fork 子进程）
  ② 子进程扫描当前内存，但不再是生成压缩的命令，而是生成 RDB 格式的快照
  ③ RDB 快照写入 AOF 文件头部，后续新命令持续追加 AOF
  ④ 加载时：先快速加载 RDB 头部（二进制，秒级）→ 再回放少量增量 AOF

  效果：RDB 的恢复速度 + AOF 的数据安全性 = 混合持久化
  
```

---

## 混合持久化:

## 第1步：子进程生成 RDB 格式的快照

### 传统 AOF 重写怎么做？
- 子进程遍历整个数据库的所有键值对，把它转换成“重建这个键所需的命令”。  
  例如内存中有 `user:1 → "Alice"`，子进程会生成：
  ```
  SET user:1 "Alice"
  ```
  然后把这条命令写入新 AOF 文件。

### 混合持久化改了哪里？
子进程不再生成**命令**，而是直接生成**RDB 格式的二进制数据**。:rocket:

- RDB 是 Redis 的内存快照格式（二进制），它直接把键值对**序列化成一串紧凑的字节**，不是可读的命令。
- 子进程做的事本质就是一次内存快照，和 `BGSAVE` 生成 RDB 文件的过程几乎相同。

**所以第1步的意思是：**  
子进程 fork 出来后，把当前内存里的全量数据，用 RDB 的编码方式（二进制），写到一个缓冲区，准备放进新 AOF 文件的开头。

---

## 第2步：RDB 快照写入 AOF 文件头部，后续命令追加

这一步描述了**新 AOF 文件的最终结构**。

### 写入头部:rocket:
子进程把刚才生成的 RDB 二进制数据**直接写到新 AOF 文件的最前面**，作为文件内容的第一部分。

> 打个比方：新 AOF 文件是一个笔记本，子进程先在笔记本第一页贴上从内存拍下来的“照片”（RDB 二进制），这张照片已经包含了截止到 fork 那一刻的所有数据。

### 后续命令持续追加:rocket:
子进程在生成 RDB 的过程中，父进程（主进程）还在处理客户端请求，可能会有新的写操作（如 `SET age 20`）。这些新命令**不能**由子进程写入（子进程的内存是 fork 时的快照，看不到新数据）。

因此，父进程会把这些新命令**同时缓存到一个缓冲区**，等子进程完成 RDB 写入后，父进程会把缓冲区的命令**追加到新 AOF 文件的后面**（紧接着 RDB 数据）。

此后，所有新的写命令也持续追加到这个文件末尾，就像普通 AOF 一样。

### 最终文件结构:rocket:
```
[RDB 二进制数据] [后续 AOF 命令1] [后续 AOF 命令2] ...
 ↑                ↑
 子进程写入      父进程写入
```

---

## 为什么这样做能加速加载？回到第3步

加载时 Redis 是这样读这个文件的：

1. **先快速加载 RDB 头部**  
   Redis 发现文件开头是 RDB 格式（有一个固定的魔数头部标识），就直接用 RDB 的解析器把数据加载到内存。  
   RDB 是紧凑二进制，解析速度极快（通常是 AOF 命令回放的 5~10 倍），这一步就能恢复**绝大部分**数据。

2. **再回放增量 AOF 命令**  
   从 RDB 数据结束的位置开始，后面的都是文本协议的 AOF 命令。这些命令数量很少（仅从子进程 fork 到宕机之间的写操作），逐条执行即可完成数据同步。

**对比纯 AOF 重写：**  
纯 AOF 文件从头到尾都是命令，加载时必须逐行解析并执行所有命令，即使数据量不大也要跑完整个文件。混合持久化用 RDB 跳过了大部分命令的重放，所以恢复速度大幅提升。

---

## 直观总结

- **第②步：** 子进程把内存数据拍成一张“二进制快照”（RDB 格式），而不是转成命令。
- **第③步：** 这张快照被放在新 AOF 文件的最前面，之后的重写期间命令和后续写命令以普通 AOF 格式追加在尾部。
- **结果：** 一个文件包含两种格式，前半部分高速加载，后半部分补充最新数据，兼顾了恢复速度和数据安全性。





### 6.2 持久化时对过期键的处理

#### RDB 文件生成阶段（BGSAVE 子进程写盘时）

```
子进程通过 fork + COW 获得了一份"快照时刻"的内存视图。
当子进程遍历所有 key 写入 RDB 文件时：

  ┌──────────────────────────────────────────────┐
  │ 子进程遍历所有 key                            │
  │                                              │
  │ 遇到一个 key → 检查是否在 expires dict 中     │
  │   ├─ 没有 TTL → 正常写入 RDB                │
  │   └─ 有 TTL ↓                               │
  │       ├─ 已过期 → 跳过！不写入 RDB！          │
  │       └─ 未过期 → 写入 RDB（key + value + TTL）│
  │           （TTL 存的是"绝对过期时间戳"）      │
  └──────────────────────────────────────────────┘

关键：已过期的 key 不会被写入 RDB 文件。
     正在过期的 key 子进程视角下已经是过期的（COW 前的状态）
```

#### RDB 文件加载阶段（Redis 重启恢复时）:rocket::rocket::rocket:

```
┌──────────────────────────────────────────────────────┐
│ Redis 启动 → 加载 dump.rdb                           │
│                                                      │
│ 主服务器（Master）加载 RDB：                          │
│   加载所有 key → 检查过期时间                         │
│     ├─ 未过期 → 正常加载                              │
│     └─ 已过期 → 跳过！不加载！                        │
│                                                      │
│ 从服务器（Slave）加载 RDB：                            │
│   Slave 不检查过期时间！全部加载！                     │
│   为什么？参见「11-Redis内存管理」——                   │
│   Slave 的过期由 Master 通过 DEL 命令同步控制          │
│   Slave 不主动过期，只被动接收 Master 的删除指令       │
└──────────────────────────────────────────────────────┘
```

#### AOF 文件写入阶段（主线程追加命令时）

```
┌──────────────────────────────────────────────────────┐
│ 客户端执行：GET expired_key                           │
│   → Redis 惰性删除检查 → 发现已过期                   │
│   → 删除该 key                                       │
│                                                      │
│ AOF 中不知道"这个 key 过期了"——因为过期不是一条命令。  │
│ Redis 需要主动往 AOF 中追加一条 DEL 命令！             │
│                                                      │
│ AOF 追加逻辑：                                        │
│   当某个 key 因为过期而被删除时（惰性删除），          │
│   Redis 会自动在 AOF 文件中追加一条：                  │
│                                                      │
│     DEL expired_key                                  │
│                                                      │
│ 这样 AOF 加载时执行到这条 DEL，就知道这个 key 被删了。  │
│ 如果不加这条 DEL → 加载 AOF 时会错误恢复过期的 key     │
└──────────────────────────────────────────────────────┘
```

#### AOF 重写阶段（BGREWRITEAOF 子进程扫描时）

```
┌──────────────────────────────────────────────────────┐
│ AOF 重写子进程（fork + COW）：                        │
│                                                      │
│ 子进程扫描当前内存中的所有 key：                       │
│   ├─ 有 TTL 且已过期 → 跳过！不生成任何命令！          │
│   └─ 未过期（或无 TTL） → 生成对应的 SET 命令          │
│                                                      │
│ 效果：AOF 重写后的文件不包含任何已过期的 key。         │
│       这和 RDB 的生成逻辑一致——已过期的 key 不写盘。   │
└──────────────────────────────────────────────────────┘
```

#### 总结——过期键在两种持久化中的处理

```
┌──────────────┬─────────────────┬──────────────────────┐
│              │   RDB           │   AOF                │
├──────────────┼─────────────────┼──────────────────────┤
│ 生成/写入阶段 │ 已过期 key 不写  │ 惰性删除触发时追加   │
│              │                 │ DEL 命令到 AOF        │
├──────────────┼─────────────────┼──────────────────────┤
│ 重写阶段     │ (同上)          │ 子进程扫描时跳过过期   │
│              │                 │ key，不生成对应命令   │
├──────────────┼─────────────────┼──────────────────────┤
│ 加载阶段     │ Master：跳过已   │ 加载全部命令，非过期  │
│              │ 过期的key       │ key 不被误删           │
│              │ Slave：全部加载  │                      │
│              │ (由Master控制)   │                      │
└──────────────┴─────────────────┴──────────────────────┘
```

## ⭐️ 面试题汇总

**Q1: BGSAVE 和 BGREWRITEAOF 都依赖 fork()。fork() 做了什么？为什么它会导致阻塞？**

> fork() 创建一个子进程——操作系统拷贝父进程的页表给子进程。页表 = 虚拟地址到物理地址的翻译字典，大小 = 内存 / 4KB × 8 字节。10GB 实例的页表约 21MB——拷贝需要 ~100ms。拷贝期间主线程被挂起——这就是 fork 的阻塞来源。内存越大 → 页表越大 → 阻塞时间越长。

**Q2: Copy-on-Write（COW）是什么？什么时候触发？**

> fork 后父子共享物理内存。父进程修改数据时 → CPU 检测到共享页被写 → 触发页故障 → OS 复制该 4KB 页 → 父进程在新页上改 → 子进程仍看原页。COW 额外内存 = 被修改的页数 × 4KB。写密集型系统在 BGSAVE 期间 COW 可能消耗 10%-50% 额外内存。

**Q3: 为什么生产环境必须关闭 THP？**

> THP 把标准 4KB 页合并成 2MB 大页。COW 以页为复制粒度——关闭 THP 时复制 4KB，开启 THP 时复制 2MB → 膨胀 512 倍。BGSAVE/AOF 重写期间，大量写操作的 COW 开销被 THP 放大 → 物理内存瞬间耗尽 → OOM。**必须 `echo never > /sys/kernel/mm/transparent_hugepage/enabled`。**

**Q4: 你如何规划 Redis 实例的内存大小和生产配置？**

> ① 单实例 ≤ 10-16GB（fork 阻塞可控）；② 物理内存 ≥ maxmemory × 1.5（COW 缓冲区）；③ 关闭 THP；④ 监控 fork 耗时（`INFO stats` 的 `latest_fork_usec` > 1 秒要告警）；⑤ 如果内存 > 16GB，考虑 Redis Cluster 拆分。

**Q5: AOF 重写为什么要 fork 而不是主线程自己做？**

> 和 RDB 一样的原因——主线程不能干慢活。AOF 重写需要扫描所有内存数据生成等价命令，10GB 数据可能需要几秒——主线程做这个会完全卡住所有客户端请求。fork 子进程做，主线程只在 fork 瞬间阻塞（~100ms），之后继续服务。这也是为什么 AOF 重写受 COW 和 THP 影响的根本原因。

**Q6: everysec 策略下最多丢失多少数据？为什么？**

> 最多丢 1 秒。因为 Redis 后台线程 B IO-2 每秒执行一次 fsync——把这一秒积累在 aof_buf 中的命令刷盘。如果恰好在这秒间隔内断电，aof_buf 中的内容还没被刷到磁盘上。不是丢 2 秒——只有 aof_buf 中的数据会丢，上一秒已 fsync 的数据在磁盘上。
