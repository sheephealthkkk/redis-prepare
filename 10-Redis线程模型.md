## 前言：最大的误解——"Redis 是单线程的"

这是 Redis 面试中最常见的误解。正确的说法是：

> **"Redis 的「命令执行」是单线程的，但 Redis 整体从来不是单线程的。"**

```
Redis 的真实线程架构（截止 Redis 7.x）：

┌─────────────────────────────────────────────────────────────┐
│                       Redis 进程                             │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │              主线程（命令执行线程）                      │  │
│  │  ① epoll 等待事件                                     │  │
│  │  ② 读取客户端请求                                      │  │
│  │  ③ 解析 + 执行命令（单线程！串行！）                     │  │
│  │  ④ 写回响应给客户端                                    │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │

│                                                             │
```

## 第一章：Redis 主线程——事件驱动的单线程引擎

### 1.1 主线程在做什么？

Redis 启动后，主线程进入一个**无限循环**：

```
while (事件循环在跑) {
    // ① 调用 epoll_wait()，等待客户端连接的 socket 变成"可读"或"可写"
    events = aeApiPoll(timeout);

    // ② 遍历所有就绪的事件
    for (每个就绪的 socket) {
        if (是新的客户端连接) {
            接受连接，把新 socket 加入 epoll 监听;
        }
        if (是已有客户端的读事件) {
            读数据 → 解析 RESP 协议 → 得到一条 Redis 命令;
            执行命令（修改内存数据）;
            把响应内容写入客户端的发送缓冲区;
            把该 socket 注册为"可写"事件;
        }
        if (是已有客户端的写事件) {
            socket 变为可写 → 把发送缓冲区的内容写出去;
        }
    }

    // ③ 执行定时任务
    processTimeEvents();  // 过期 key 清理、serverCron 定时任务等
}
```

**这个循环就是 Redis 的"心跳"。** 主线程的角色就是在这一个循环中处理所有客户端的所有读写和执行操作。

### 1.2 Redis 自己实现了事件库——ae

Redis 没有使用 libevent/libuv 这些通用事件库，而是自己写了一个极简的事件驱动库叫 `ae`：

```
ae 支持的四种底层 I/O 多路复用：

  平台         │ 底层 API   │ 选项
  ─────────────┼───────────┼──────
  Linux        │ epoll     │ AE_EPOLL    ← 生产环境主力
  macOS/BSD    │ kqueue    │ AE_KQUEUE
  Solaris      │ evport    │ AE_EVPORT
  兜底         │ select    │ AE_SELECT   ← 最慢，但哪都能跑
```

### 1.3 epoll 工作原理——为什么单线程能扛数十万连接

**epoll 是 Linux 内核提供的高效 I/O 通知机制。** 用生活类比：

```
select 的做法（Redis 也可以用它做兜底）：
  老板每隔一秒出门巡视：员工 A 有没有活？员工 B 有没有活？员工 C...
  即使 100 个员工只有 1 个在干活，老板还是要把 100 个都问一遍
  → 时间复杂度 O(N)，连接数越多越慢

epoll 的做法（Redis 默认）：
  老板在办公室喝茶。哪个员工有活了，按一下铃，老板再走过去看。
  只有 1 个员工有活 → 只处理 1 个 → 其他 99 个完全不管
  → 事件复杂度 O(1)（准确说 O(活跃连接数)，而非 O(总连接数)）
```

**epoll 的底层机制：**

### 核心部件与职责

#### 1. `eventpoll` (事件池) —— “智能仓库总部”

这是 `epoll_create` 创建出来的**大总管对象**。它手里有三样法宝，各司其职：

- **红黑树 (`rbr`)**：**“在途包裹登记簿”**
  - **作用**：存放你**所有想要监控的快递单号 (fd)**。
  - **为什么需要它**：当你想新增、删除或修改一个监控对象时，它能极快地完成查找和操作。就像一个索引清晰的登记簿，能快速定位到某一个包裹的信息，而不需要一页一页翻遍整个簿子。
- **就绪链表 (`rdllist`)**：**“到达通知看板”**:o:
  - **作用**：一个列表，上面只挂着那些**已经到货的快递单号**。
  - **为什么需要它**：当你来仓库问“有谁到了？”时，大总管不需要去成百上千个登记在册的包裹中挨个确认，只需要把手边这块看板上挂着的几个单号直接给你就行。这就是 epoll “不用遍历全部连接”的秘密。
- **等待队列 (`wq`)**：**“你的专属休息室”**
  - **作用**：当你来仓库查询，但看板还是空的（没有任何快递到达），你就进这个休息室**休眠**。
  - **为什么需要它**：:o::o::o::o::o::o::o::o::o::o::o::o::o:把你休眠在这里，系统可以在有快递到达时精准地叫醒你，而不必你自己不停地跑来跑去问，节省系统资源。

#### 2`epitem` (事件项) —— “快递追踪卡”

当你对一个连接（socket）说“帮我盯着它”，`epoll_ctl` 就会生成这张**追踪卡**。

- **它身上的关键信息**：
  - **指着具体的快递 (sockfd)**：标记这张卡是为哪个包裹建立的。
  - **记着你的要求 (event)**：比如你关心“包裹到达”(读事件)还是“可以寄出”(写事件)。
  - **一个挂在“登记簿”上**：这张卡本身，就插在仓库总部的红黑树登记簿里，证明这个包裹正被监控。
  - **一个挂在“通知看板”上**：当包裹真正到达时，这张卡就会被临时挂到就绪链表看板上，让你能看见它。

**关键理解**：`epitem` 就是连接“被监控的 socket”和“epoll 实例”之间的桥梁。:o:

#### 3. `ep_poll_callback` (回调函数) —— “到货自动通知铃”

这是整个机制**最画龙点睛的部分**，它是一段预先写好的程序。

- **它被安插在哪里？**
  它被粘贴在快递包裹本身的物流系统上，也就是**socket 文件描述符的等待队列**里。
- **它的作用是什么？**
  当包裹真的到达（网卡收到数据，内核把数据放进 socket 缓冲区），物流系统就会自动触发这个铃。
  - **铃响了以后它会做什么？**:o::o:
    1. 它不假思索地拿起代表这个包裹的 **`epitem` (快递追踪卡)**。
    2. 把它“啪”的一下，贴到仓库总部的 **“到达通知看板” (`rdllist`)** 上。
    3. 然后，去 **“你的专属休息室” (`wq`)** 把你拍醒（如果当时你正在休眠等待）。
- **为什么这是关键？**
  因为这个通知是 socket 在数据到达时**自己主动发出的**。它让你完全不用去轮询，“你等它”变成了“它叫你”。

### 把它们串起来，讲一个完整的故事

1. **你创建了仓库总部 (`epoll_create`)**，拿到了看板、登记簿和休息室的钥匙。
2. **你告诉仓库你要跟踪包裹A (`epoll_ctl ADD`)**：
   - 仓库制作了一张“包裹A追踪卡” (`epitem`)。
   - 把卡存档在**登记簿（红黑树）**里。
   - 在包裹A的物流流程上，设好了一个**自动通知铃 (`ep_poll_callback`)**。这个铃的动作就是：一旦包裹到达，就把追踪卡贴到看板上，并来休息室叫你。
3. **你开始等待 (`epoll_wait`)**：
   - 你问仓库：“看板上有卡吗？”
   - **如果没有**：你就在休息室 (`wq`) 里睡觉。
   - **如果有**：仓库直接把看板上挂着的追踪卡都摘下来给你。你就能立刻知道该去处理哪个包裹了。
4. **快递到达，魔法发生**：
   - 包裹A的数据来了，物流系统触发了那个**自动通知铃 (`ep_poll_callback`)**。
   - 这个铃把包裹A的追踪卡 (`epitem`) 从登记簿里拿出来（逻辑上是复制引用），挂到了**看板 (`rdllist`)** 上。
   - 然后跑来休息室 (`wq`)，把你叫醒。
   - 你被叫醒，结束 `epoll_wait`，回去一看手上多了一份清单，上面就写着“包裹A到了”。你开始处理包裹A。

- **红黑树** = 为了快速增删改查而存在的“全量监控清单”。
- **就绪链表** = 只展示活跃连接的“高效交付窗口”。
- **等待队列** = 让无事可做的等待者精确休眠的“节能装置”。
- **epitem** = 维系连接和监控关系的“身份卡片”。
- **回调函数** = 把“主动轮询”转变为“被动通知”的“魔法开关”。



好的，我们把之前聊的 epoll 机制和 Redis 多线程 IO 串起来，用一个完整的 **`GET name` 请求** 的生命周期来走一遍。

假设一个客户端已经和 Redis 建立了 TCP 连接，现在它发送了 `GET name` 命令。我们从操作系统内核收到数据包开始，一路追到响应发回客户端。

---

### 第一阶段：数据到达与内核通知 (epoll 登场)

1.  **网卡收到数据包，内核协议栈处理**
    客户端的 `GET name` 数据包到达服务器网卡，通过 DMA、硬中断、软中断，最终 TCP 协议栈将数据放入属于这个客户端 socket 的**接收缓冲区**。

2.  **触发 epoll 回调，点亮“通知看板”**
    数据进入 socket 缓冲区这个动作，会触发我们之前在 socket 等待队列里挂载的 **`ep_poll_callback` 回调函数**。
    这个回调函数会找到代表这个 socket 的 **`epitem`**，把它挂到 `eventpoll` 结构的**就绪链表 (`rdllist`)** 上。此时，这个客户端 socket 的状态在 epoll 眼里就变成了“可读”。

3.  **唤醒 Redis 主线程**
    `ep_poll_callback` 接着检查 `eventpoll` 的等待队列，发现 Redis 的主线程正在那里休眠（因为之前调用 `epoll_wait` 时没有事件），于是把它唤醒。

---

### 第二阶段：主线程调度与 IO 线程搬运 (多线程 IO)

4.  **主线程从 epoll_wait 返回**
    被唤醒的 Redis 主线程从 `epoll_wait` 调用中返回，拿到一个就绪事件列表，里面包含了这个客户端 socket 的文件描述符。

5.  **主线程分配读任务**
    主线程并不会自己在这个 socket 上执行 `read`。它把这个可读的 socket，连同其他一批可读的 socket，**轮询分配给 IO 线程池**。比如，把这个 socket 分配给了 IO 线程 1。

6.  **IO 线程并行搬运数据**
    IO 线程 1 对自己负责的这批 socket 执行 `read` 系统调用。对于我们的客户端 socket，IO 线程 1 将数据从内核的接收缓冲区，搬运到 Redis 为该客户端维护的**输入缓冲区 (`querybuf`)** 里。此时 `querybuf` 里有了 `*2\r\n$3\r\nGET\r\n$4\r\nname\r\n` 这样的 RESP 协议文本。

7.  **主线程自旋等待读完成**
    主线程通过忙等（自旋），等待所有 IO 线程完成本轮读任务。

---

### 第三阶段：单线程执行命令 (Redis 核心)

8.  **主线程解析并执行命令**
    所有数据读完后，主线程开始单线程地干活：
    -   遍历所有本轮读到了数据的客户端。
    -   从我们的客户端 `querybuf` 中解析出 `GET` 和 `name`。
    -   去全局的键值对字典 (`dict`) 里查找 `name` 这个键，拿到它的值，比如 `"Alice"`。
    -   将响应 `"Alice"` 格式化成 RESP 协议 `$5\r\nAlice\r\n`，追加到客户端的**输出缓冲区 (`buf` / `reply list`)** 里。

---

### 第四阶段：IO 线程再次出动，写回响应

9.  **主线程分配写任务**
    命令执行完了，客户端有了需要发送的响应数据，状态变为“可写”。主线程再次将这类客户端分配给 IO 线程池。

10. **IO 线程并行写回数据**
    分配给这个客户端的 IO 线程（比如还是 IO 线程 1），调用 `write` 系统调用，拼命把输出缓冲区里的 `$5\r\nAlice\r\n` 通过网络发出去。这个过程是非阻塞的，一次写不完就下次再写。

11. **主线程自旋等待写完成**
    主线程再次忙等，直到所有写任务完成。

---

### 第五阶段：善后与新一轮等待

12. **主线程收尾**
    主线程对这些客户端进行收尾工作，比如检查是否需要释放临时资源。

13. **主线程修改 epoll 监听事件**
    对于一次没写完的客户端，主线程会通过 `epoll_ctl` 修改这个 socket 的监听事件，添加上 `EPOLLOUT`（可写事件），这样内核会在 socket 再次可写时通知我们。

14. **重回 epoll_wait**
    所有事做完，主线程再次调用 `epoll_wait`，进入休眠，等待内核通知下一批就绪事件。

---

### 一张图总结全流程

```text
[网卡] → [内核接收缓冲区]
                │
         触发 ep_poll_callback
                │
          挂入 rdllist，唤醒 Redis 主线程
                │
        ==============================
               Redis 主线程
        ==============================
                │
    ┌───────────┼───────────┐
    │ epoll_wait 返回       │
    │ 拿到可读 socket 列表   │
    │ 分配给 IO 线程池       │
    └───────────┼───────────┘
                │
    ┌───────────┼───────────┐
    │ IO 线程并行 read()    │  ← 搬运到 querybuf
    │ 主线程自旋等待        │
    └───────────┼───────────┘
                │
    ┌───────────┼───────────┐
    │ 主线程单线程执行:      │
    │ 解析协议 → 执行GET    │  ← 纯内存操作
    │ → 生成响应放入输出缓冲  │
    └───────────┼───────────┘
                │
    ┌───────────┼───────────┐
    │ IO 线程并行 write()   │  ← 从输出缓冲区发出
    │ 主线程自旋等待        │
    └───────────┼───────────┘
                │
         主线程收尾，重回 epoll_wait 休眠
```

**核心分工总结：**
-   **epoll**：负责告诉 Redis “谁可以读/写了”，让 Redis 只关注活跃连接。
-   **IO 线程**：负责数据的网络搬运（socket ↔ 缓冲区），并行处理，省力气。
-   **主线程**：只干两件事——调度 IO 线程，以及**单线程**执行所有命令逻辑。保证了高性能，也保证了原子性。

**这就是 Redis 单线程能同时处理数万连接的秘密所在**——不是 Redis 自己去管理这么多连接，而是 **Linux 内核（epoll）替它管理，Redis 只需要处理"有活干"的那几个连接。**

### 1.4 为什么 Redis 不直接把 epoll 换成多线程？

**因为命令执行不需要多线程。** 这是关键推导：

```
Redis 命令执行的特点：
  SET key "hello"        → 1 次内存分配 + 1 次 dict 查找 + 1 次内存写入
                           耗时：~1 微秒

  INCR counter           → 1 次 dict 查找 + 1 次整数运算
                           耗时：~0.5 微秒

  LPUSH list "a"         → 1 次 quicklist 操作
                           耗时：~2 微秒

  结论：绝大多数 Redis 命令执行时间都是微秒级的。
        单线程 1 微秒处理 1 个命令 → 1 秒处理 100 万个命令。
        瓶颈根本不在 CPU！瓶颈在网络带宽和内存带宽。
```

**所以，Redis 的瓶颈是 I/O，不是 CPU。** 既然 CPU 不是瓶颈，搞多线程就是无意义的——线程本身会引入锁、上下文切换、竞态等额外开销。;o::o::o::o::o::o::o:

## 第二章：Redis 6.0 之前为什么不使用多线程？

### 2.1 核心原因——多线程解决不了 Redis 的瓶颈

```
Redis 的性能瓶颈到底在哪？

  假设单机 QPS 10 万，每个请求的耗时分布：
    ├─ 网络 I/O（客户端到 Redis 的 RTT）：约 0.1-1ms    ← 占 99%！
    ├─ epoll 等待：约 0.01-0.1ms
    ├─ 读 socket 缓冲区：约 0.001ms（DMA 传输，CPU 不参与）
    ├─ 解析 RESP 协议：约 0.001ms
    ├─ 执行命令：约 0.0005ms（0.5 微秒）                ← CPU 仅占 0.05%！
    ├─ 构造响应：约 0.001ms
    └─ 写 socket 缓冲区：约 0.001ms

  结论：99%+ 的时间都在等网络，不在算数据。
        多线程加速计算（执行命令）→ 加速 0.05% 的部分 → 整体性能几乎不变。
```

### 2.2 次要原因——单线程的优势

```
单线程天然附带的好处：
  ① 零锁开销——不需要 Mutex、不需要 CAS、不需要 synchronized
  ② 零上下文切换——没有线程调度、没有 CPU 缓存污染
  ③ 零竞态条件——同一个 key 不可能被两个线程同时修改
  ④ 代码极简——不用考虑并发安全，所有数据结构的实现都是单线程假设
  ⑤ 便于调试——GDB 调试时没有多线程的复杂度
```

**用一句话总结：** 在 6.0 之前，引入多线程的额外复杂度 > 带来的性能收益。不划算。

## 第三章：Redis 6.0 之后为何引入多线程 I/O？

### 3.1 推动力——网络带宽追上来了

```
10 年前的瓶颈：CPU 算力不足 → 多线程有意义
今天的瓶颈：网络带宽 → 从 1Gbps 升级到 10Gbps、25Gbps、100Gbps

万兆网卡普及后：
  Redis 单线程每秒能执行约 100 万次内存操作（微秒级）
  但万兆网卡每秒能传输的数据量约 1.2 GB
  每个请求平均 1KB → 网络每秒能传输约 100 万次请求

  瓶颈从 CPU 转向了网络！而现在网络带宽又大到了"单线程处理网络 I/O 忙不过来"的程度
```

### 3.2 6.0 版本做了什么——用多线程帮主线程做网络读写

### 3.3 为什么命令执行依然保持单线程？

即使是 6.0，**命令执行依然在主线程串行**。这是为了保持 Redis 的核心语义：
- INCR 的原子性 = 单线程执行
- LPUSH + BRPOP 的事务性 = 单线程执行
- 所有数据结构的无锁设计 = 单线程执行

**如果连命令执行都搞多线程，Redis 就变成另外一个 Redis 了**——所有数据结构都要加锁，复杂度爆炸，微秒级的命令执行变成毫秒级的锁等待，反而更慢。

### 3.4 配置与启用

```bash
# redis.conf

# ① 开启多线程 I/O
io-threads 4                    # IO 线程数（建议 = CPU 核数，不超过 8）
                                # 默认 1 = 单线程（不开启多线程 I/O）

# ② 是否把"读"也交给 IO 线程（默认只把"写"给 IO 线程）
io-threads-do-reads yes         # yes = 读写都用多线程
                                # no  = 只用在写（写的数据量比读大）
```

**建议配置：**
```
4 核以下：io-threads 1（单线程，多线程 I/O 没有收益）
4 核：io-threads 2 或 3
8 核：io-threads 4
8 核以上：io-threads 4-8（不要超过 8，线程太多调度开销更大）

什么时候需要开？
  QPS < 5w/s  ：不用开（单线程完全够）
  QPS 5-10w/s ：建议开 2-4
  QPS > 10w/s ：建议开 4-8
  但前提是你的网络带宽够大（10Gbps+）
```

## 第四章：Redis 后台线程——除了主线程还有什么？

### 4.1 三兄弟——BIO 线程池

Redis 从 4.0 开始有 **BIO（Background I/O）线程**，共 3 个后台线程，处理主线程不想干的慢活：

```
┌──────────────────────────────────────────┐
│           Redis 的 3 个后台线程            │
├──────────┬───────────┬───────────────────┤
│ BIO-1    │ BIO-2     │ BIO-3             │
│ 文件关闭  │ AOF fsync │ 惰性释放内存       │
│          │           │ (lazy free)       │
└──────────┴───────────┴───────────────────┘

为什么需要他们？

  BIO-1 (close file)：
    主线程执行 BGSAVE 或 AOF 重写后，子进程产生的临时文件需要关闭
    close() 系统调用可能阻塞（文件系统刷盘）→ 不能卡主线程 → 交给后台线程

  BIO-2 (AOF fsync)：
    主线程把 AOF 数据写入 aof_buf 后，fsync（刷盘）的任务由这个线程执行
    fsync 是磁盘 I/O → 需要几毫秒 → 不能卡主线程 → 交给后台线程

  BIO-3 (lazy free)：
    执行 UNLINK 删除大 key 时 → 主线程只把 key 从键空间摘掉
    释放大量内存（遍历数百万 dictEntry 逐个 free）由这个线程执行
    → 主线程不阻塞 → 用户完全感知不到
```

### 4.2 Lazy Free 异步删除——你执行 UNLINK 时背后发生了什么

```bash
# 一个大 Set 有 500 万个元素
127.0.0.1:6379> SCARD big:set
(integer) 5000000

# DEL：主线程删除 → 遍历 500 万个 dictEntry → 阻塞 200ms
127.0.0.1:6379> DEL big:set     # ← 其他客户端卡了 200ms！
(integer) 1

# UNLINK：主线程只摘掉键空间引用 → 后台线程慢慢释放 → 立即返回
127.0.0.1:6379> UNLINK big:set  # ← 微秒级返回！
(integer) 1
# 此时后台 BIO-3 线程在跑：把 500 万个 dictEntry 的 free 逐个做完
```

**Lazy Free 的配置：**

```bash
# redis.conf

# ① DEL 命令是否自动转为 UNLINK？（默认 no，保持 DEL 语义）
lazyfree-lazy-eviction no       # 内存淘汰时的删除 → 是否异步
lazyfree-lazy-expire no         # key 过期时的删除 → 是否异步
lazyfree-lazy-server-del no     # 服务端内部删除（如 RENAME） → 是否异步
replica-lazy-flush no           # 从节点 flush → 是否异步

# 如果全部设为 yes → 所有删除都异步 → 主线程永不阻塞
# 但代价是内存释放不及时（后台线程忙的时候，内存占着没放）
```

### 4.3 BGSAVE / BGREWRITEAOF —— 子进程（不是线程）

```
严格区分：
  后台线程 = 同一个进程内的多个线程，共享内存（但有锁保护）
  子进程   = fork() 出来的独立进程，不共享内存（靠 COW 机制）

BGSAVE：
  主线程 fork() ──▶ 子进程持有自己独立的内存页表
                  (共享主进程的数据，改内存时 COW 复制)
                  子进程把数据写入 dump.rdb
                  期间主线程继续处理请求

BGREWRITEAOF：
  主线程 fork() ──▶ 子进程持有自己独立的内存页表
                  子进程扫描当前内存数据，生成新的 AOF 文件
                  期间主线程的写命令记录在 AOF 重写缓冲区中
```

## ⭐️ 面试题汇总

**Q1: Redis 是单线程的吗？**

> 不完全正确。Redis 的**命令执行**是单线程的（一直保持），但 Redis 整体有多线程：① Redis 4.0+ 的 BIO 后台线程（3 个，关闭文件/AOF fsync/惰性释放）；② Redis 6.0+ 的 I/O 线程组（网络读写多线程，默认关闭）；③ BGSAVE/BGREWRITEAOF 是 fork 出来的独立子进程。所以正确的是：「Redis 的核心命令执行是单线程的。」

**Q2: Redis 6.0 之前为什么不使用多线程？**

> 瓶颈不在 CPU 而在网络 I/O。Redis 命令执行通常是微秒级——单线程每秒能处理几十万次。用多线程执行命令 = 加锁 + 上下文切换的开销 > 收益。同时单线程天然避免锁竞争、上下文切换、竞态条件，代码简单可靠。

**Q3: Redis 6.0 之后为什么又引入了多线程？**

> 网络带宽从 1Gbps 升级到 10Gbps+ → 瓶颈真正转移到了网络 I/O。6.0 引入的多线程仅用于**网络读写**（socket read/write），命令执行仍保持单线程。相当于只把最耗时的网络部分多线程化了，核心的原子性不变。

**Q4: epoll 为什么比 select/poll 快？**

> select/poll 每次调用都要全量遍历所有 fd——时间复杂度 O(N)，连接越多越慢。epoll 使用事件通知——内核在数据到达时通过回调把 fd 加入"就绪链表"，epoll_wait 只检查这个链表——O(1)。同时 epoll 只在注册时拷贝一次 fd，select 每次调用都要全量拷贝。

**Q5: Redis 有哪些后台线程？分别做什么？**

> BIO-1：关闭文件（BGSAVE/AOF 重写后的临时文件 close）；BIO-2：AOF fsync（每秒把 AOF 缓冲区刷到磁盘）；BIO-3：惰性释放内存（UNLINK 异步删除大 key）。这三个线程让主线程不需要等待慢操作（磁盘 I/O、大量内存 free）。

**Q6: UNLINK 和 DEL 的区别是什么？底层发生了什么？**

> DEL：主线程同步删除——遍历所有 dictEntry 逐个 free 内存——大 key 删除可能阻塞几百毫秒。UNLINK：主线程只把 key 从全局键空间摘掉（微秒级），真正的内存释放交给 BIO-3 后台线程异步完成——主线程不阻塞。
