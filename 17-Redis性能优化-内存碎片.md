# Redis 性能优化 —— 内存碎片

---

## 第一章：什么是内存碎片？

### 1.1 用生活类比理解

```
搬家类比：

  你有一个 100 立方米的仓库，陆陆续续存了 50 个箱子。

  情况 1（无碎片）：
    箱子整齐排列 → 用了 50m³，空闲 50m³ 是一整块
    → 可以再放进一个 50m³ 的大箱子 ✓

  情况 2（有碎片）：
    箱子东一个西一个 → 用了 50m³，但空闲的 50m³ 是零散的
    → 虽然总空闲 50m³，但最大的连续空间只有 5m³
    → 想放一个 10m³ 的箱子 → 放不下！即使总空闲够用 ✗
```

**Redis 的内存碎片就是：** 虽然 Redis 里还有空闲内存，但这些空闲空间是零散的——想分配一块大内存时找不到"连续"的空间。

### 1.2 碎片是怎么产生的？:rocket::rocket::rocket::rocket::rocket:

```
产生碎片的三个主要原因：

原因 1：频繁修改不同类型的数据
  SET key1 "hello"   → 分配 64B 块
  DEL key1            → 释放 64B 块
  SET key2 "world!!"  → 需要 128B 块
  → 但 64B 块的位置在中间 → jemalloc 无法把它和周围的空闲合并
  → 形成碎片

原因 2：不同大小的分配混在一起
  Redis 频繁分配和释放不同大小的内存块 → 产生"空洞"

原因 3：jemalloc 的固定块大小机制
  jemalloc 用固定块大小分配（8,16,32,64,128,256,512...字节）
  申请 50B → 分配 64B 块 → 14B 的"内碎片"（块内的浪费）
```

mysql磁盘层面的

### MySQL 磁盘碎片的产生（以 InnoDB 为例）

MySQL 的数据和索引都存储在 **数据页**（默认 16KB）里，页在磁盘上是连续分配的。但表的数据经常变动：

- **可变长度字段更新**：比如 `VARCHAR(255)` 的字段，原来存 `'Alice'`（5 字节），后来更新成 `'Alexander'`（9 字节）。
- **行变长 → 页放不下**：InnoDB 会尝试在同一个页内移动行，但如果页内剩余空间不够，就会发生**页分裂**：
  - 原来的行被标记删除，新行写到另一个有足够空间的页。
  - 原页留下一个“空洞”（碎片）。
- **删除操作**：删除一行不会立即回收空间，只是标记为删除，导致页内产生碎片。

### 碎片对 MySQL 性能的核心影响：随机 I/O 增多

MySQL 查询数据时，理想情况是：

- 索引定位到某个页 → 把连续的页读到内存（顺序读）。
- 因为数据紧凑，一个页里有很多行，一次 I/O 能载入大量相关数据。

碎片化后：

- **逻辑上相邻的行物理上分散**：索引的叶子结点顺序还是 1→2→3→4，但物理页可能东一块西一块。
- 范围查询（`SELECT * FROM t WHERE id BETWEEN 100 AND 200`）原本可能只需读几个连续的页，现在需要跳来跳去读很多不连续的页，**随机读大幅增加**。
- 随机读比顺序读慢几十倍（机械硬盘）甚至几倍（SSD），严重影响吞吐量。
- **页利用率低**：缓冲池（Buffer Pool）里载入的每一页都只含少量有效数据，浪费内存，间接增加 I/O 压力。

| Redis 内存碎片 | MySQL 磁盘碎片                         |                                               |
| :------------- | :------------------------------------- | --------------------------------------------- |
| **所在位置**   | 内存（RAM）                            | 磁盘（HDD/SSD）                               |
| **产生原因**   | 频繁修改不同大小的数据、内存分配器行为 | 可变长度字段更新、页分裂、删除                |
| **直接后果**   | 内存利用率下降，申请不到大块连续内存   | 磁盘空间浪费，查询随机 I/O 增加，缓冲池效率低 |
| **解决方式**   | 重启 Redis、`MEMORY PURGE`、换分配器   | `OPTIMIZE TABLE`、重建表、用合适字段类型      |



### 1.3 如何查看碎片率？

```bash
# ① 查看内存信息
127.0.0.1:6379> INFO memory
# used_memory:          8589934592  (8GB)  ← Redis 实际管理的数据量
# used_memory_rss:      12884901888 (12GB) ← 操作系统看到 Redis 占用的物理内存
# mem_fragmentation_ratio: 1.50           ← 碎片率 = RSS / used_memory

# 碎片率计算：
#   1.0  = 完美（无碎片）
#   1.0~1.5 = 正常（轻微碎片，可接受）
#   > 1.5   = 碎片偏高（需要考虑优化）
#   > 2.0   = 碎片严重（建议重启或碎片整理）
#   < 1.0   = RSS < 数据量（部分数据被 swap 到磁盘——更危险！）
```

```bash
# ② 查看 jemalloc 级别的碎片统计（Redis 4.0+）
127.0.0.1:6379> MEMORY MALLOC-STATS
# 输出 jemalloc 各大小类别的分配/释放/碎片统计
```

## 第二章：如何解决内存碎片？

### 2.1 方案一：碎片自动整理（activedefrag — Redis 4.0+）

Redis 4.0 起内置了**自动碎片整理**功能——在后台把零散的数据搬到一起，腾出连续空间：

```
碎片整理的原理（搬家类比）：

  整理前（碎片化）：
    [A][ ][B][ ][C][ ][D][ ]  ← 有空隙
        ↑       ↑
       空闲     空闲

  整理后（紧凑）：
    [A][B][C][D][ ][ ][ ][ ]  ← 数据搬到了一起，空闲是连续的
              ↑               ↑ 一整块连续空间
```

```bash
# ===== 开启自动碎片整理 =====
# redis.conf

activedefrag yes                      # 开启自动碎片整理（默认关闭）

# 触发条件（三个条件都满足才启动整理）
active-defrag-ignore-bytes 100mb      # 碎片至少 100MB 才整理（太少不值得）
active-defrag-threshold-lower 10      # 碎片率 ≥ 10% 开始整理
                                      # （即 mem_fragmentation_ratio ≥ 1.1）
active-defrag-threshold-upper 100     # 碎片率 ≥ 100% 时全力整理
                                      # （即 mem_fragmentation_ratio ≥ 2.0）

# CPU 时间控制（防止碎片整理占用太多 CPU）
active-defrag-cycle-min 5             # 最少用 5% CPU 时间做整理
active-defrag-cycle-max 75            # 最多用 75% CPU 时间做整理

# 整理策略
active-defrag-max-scan-fields 1000    # 每次扫描最多 1000 个 field
```

**注意事项：**
- 碎片整理是**后台操作**，不会阻塞主线程:rocket::rocket:
- 但会消耗 CPU——碎片严重时 CPU 可能额外消耗 10-30%
- 建议先观察碎片率趋势再决定是否开启——如果碎片率一直 < 1.5，不必开

### 2.2 方案二：重启（最简单但粗暴）:rocket:

```bash
# 如果碎片率超过 2.0 且持续恶化，重启是最快的手段
# 前提：有持久化（RDB/AOF）保证数据恢复

# ① 确保 AOF/RDB 是最新的
BGSAVE

# ② 重启 Redis
redis-server stop
redis-server start

# ③ Redis 自动加载 dump.rdb 或 appendonly.aof 恢复数据
# 重启后碎片率降到 1.0
```

### 2.3 方案三：更换内存分配器（jemalloc → 其他）

```bash
# 编译 Redis 时更换分配器
make MALLOC=libc    # 使用系统默认 glibc malloc（碎片少但性能略低）
                    # 不推荐——jemalloc 性能更好
```

### 2.4 方案四：预防——从源头减少碎片产生

```
① 避免频繁创建和删除大量 key
   - 用 TTL 自动过期替代手动 DEL（删除更平滑）
   - 避免"批量创建 → 批量删除"的循环

② 控制 key 大小尽量一致
   - 大小相近的内存块分配和释放不容易产生碎片

③ 使用 UNLINK 替代 DEL（异步删除，减少主线程参与内存释放）

④ 控制编码升级
   - Hash/Set/ZSet 的 listpack/intset → dict/skiplist 升级不可逆
   - 一旦升级，内存占用大幅增加
```

## 第三章：碎片率异常排查

```
mem_fragmentation_ratio > 2.0：
  → 碎片严重 → 考虑开启 activedefrag 或重启

mem_fragmentation_ratio < 1.0：
  → RSS < 数据量 → 操作系统把部分 Redis 内存换出到了 swap！
  → Redis 访问这部分数据时会触发缺页中断→ 磁盘 I/O → 极慢
  → 排查：系统是否内存不足 → 调整 maxmemory 或加内存
```

## ⭐️ 面试题

**Q1: mem_fragmentation_ratio 是什么？正常范围是多少？**

> 碎片率 = used_memory_rss / used_memory。RSS 是操作系统看到的物理内存，used_memory 是 Redis 实际管理的数据量。正常范围：1.0~1.5。> 2.0 碎片严重需整理；< 1.0 数据被 swap 到磁盘更危险。

**Q2: 碎片率太高怎么办？**

> ① 开启 activedefrag（Redis 4.0+）自动后台碎片整理；② 碎片率 > 2 且持续恶化→ 重启（前提是有持久化保证数据恢复）；③ 长期优化：控制 key 大小均匀、避免频繁创建删除。

**Q3: 为什么 jemalloc 会产生碎片？**

> jemalloc 用固定大小的块（8/16/32/64...字节）。申请 50B → 分 64B 块 → 14B 内碎片。频繁分配释放不同大小的块 → 产生"空洞"（外碎片）。这是内存分配器的通用问题，不是 jemalloc 独有。
