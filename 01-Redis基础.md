# Redis 基础 —— 深度剖析

---

## ⭐️ 1. Redis 为什么这么快？

Redis 号称单机 QPS 可达 10w+，这个数字不是吹出来的。我们从四个维度来拆解它快在哪里：

---

### 1.1 纯内存操作（Memory-Based Storage）

**核心事实：Redis 绝大部分操作都是直接读写内存，不涉及磁盘 I/O。**

| 存储介质 | 访问延迟 | 带宽 |
|---------|---------|------|
| L1 Cache | ~1 ns | — |
| L2 Cache | ~4 ns | — |
| 主存（RAM） | ~100 ns | 20-50 GB/s |
| SSD（NVMe） | ~20-100 μs | 2-7 GB/s |
| 网络往返（同机房） | ~200-500 μs | — |

- **内存随机读延迟约 100ns，SSD 随机读约 100μs，差了 1000 倍。**
- 这也是为什么 Redis **不能当"大号 Memcached"来滥用**——内存是昂贵资源，数据量 > 内存时性能会断崖式下跌（涉及 swap / 淘汰）。
- 内存操作的瓶颈不在介质本身，而在 **CPU Cache Miss** 和 **内存分配/回收**。Redis 通过下面的数据结构设计来规避这些问题。

**面试延伸点：** 面试官可能追问"如果数据量超过物理内存怎么办？"——这引向 Redis 的内存淘汰策略（LRU/LFU）和持久化/集群话题。

---

### 1.2 高效的 I/O 模型（I/O Multiplexing 与单线程事件循环）

这是 Redis 快的最**核心**原因，也是面试高频考点。

#### 1.2.1 为什么单线程反而快？

**先破一个常见误解：Redis 不只是"单线程"。**
- Redis 6.0 之前的"单线程"仅指**命令处理**是单线程的。
- Redis 6.0+ 引入了多线程 I/O（网络读写），但命令执行仍然是单线程。

**单线程为什么反而有优势？**

```
多线程模型的问题：
┌─────────────────────────────────────┐
│ Thread-1 → [抢锁] → 修改 keyA       │
│ Thread-2 → [等待锁] → 也要修改 keyA  │
│ Thread-3 → [上下文切换] → ...        │
│                                      │
│ 代价：锁竞争 + 上下文切换 + 死锁风险  │
└─────────────────────────────────────┘

Redis 单线程模型：
┌─────────────────────────────────────┐
│ 事件循环线程：                       │
│   while(1) {                        │
│       events = aeApiPoll(...)       │
│       for each event:               │
│           processCommand(event)     │
│   }                                 │
│                                      │
│ 代价：0 锁竞争，0 上下文切换          │
└─────────────────────────────────────┘
```

**单线程快的关键前提：**
1. **CPU 不是瓶颈。** Redis 的性能瓶颈在**网络 I/O 和内存带宽**，而非 CPU 算力。单核跑满 10w QPS 的简单命令，CPU 可能才用了 30%。
2. **避免了多线程的所有开销。** 没有锁竞争、没有上下文切换、没有伪共享（false sharing）、没有竞态条件。
3. **简单的命令都是 O(1) 操作。** GET/SET/LPUSH 等命令执行时间在微秒级，单线程足够处理。

**但是，单线程的致命弱点：**
- 一个慢命令（如 KEYS *、HGETALL 大 hash、SORT）会**阻塞整个 Redis 实例**，后续所有请求排队等待。
- 这就是为什么 Redis 提供了 SCAN 系列命令、UNLINK（异步删除）等。

#### 1.2.2 I/O 多路复用（epoll / kqueue / select）

Redis 的事件处理基于 **Reactor 模式**，包装了操作系统的 I/O 多路复用机制：

```
                     ┌──────────────┐
                     │   Clients    │
                     └──┬──┬──┬────┘
                        │  │  │  socket1 readable
                        │  │  │  socket3 readable
                        ▼  ▼  ▼
                  ┌─────────────────┐
                  │  I/O Multiplex  │  ← epoll_wait()
                  │  (epoll/kqueue) │     一个线程监听数百个 socket
                  └────────┬────────┘
                           │ 就绪事件队列
                           ▼
                  ┌─────────────────┐
                  │  Event Loop     │
                  │  while(1) {     │
                  │    读取命令      │
                  │    执行命令      │
                  │    写回响应      │
                  │  }              │
                  └─────────────────┘
```

**关键组件——ae 事件库（ae.c）：**

Redis 自己实现了一个轻量事件库 `ae`，对不同平台的 I/O 多路复用做了统一封装：

| 宏 | 底层系统调用 | 平台 |
|----|------------|------|
| AE_EPOLL | epoll | Linux |
| AE_KQUEUE | kqueue | macOS/BSD |
| AE_SELECT | select | 通用兜底 |

**epoll 的核心优势（相对于 select/poll）：**

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| 数据结构 | 数组（FD_SETSIZE 限制 1024） | 链表（无限制） | 红黑树 + 就绪链表 |
| 工作模式 | 每次遍历整个集合 O(n) | 每次遍历整个集合 O(n) | 只返回就绪事件 O(1)，事件通知 |
| 数据拷贝 | 每次调用全量传入传出 | 每次调用全量传入传出 | epoll_ctl 注册一次，epoll_wait 只拷贝就绪 |
| 触发方式 | 水平触发 | 水平触发 | 水平触发 + 边缘触发（ET） |
| 连接数 1000 的耗时 | ~占用 CPU 时间正比于 n | ~占用 CPU 时间正比于 n | ~正比于活跃连接数 |

**epoll 原理简述：**
1. `epoll_create()`：内核创建 eventpoll 对象（包含红黑树 `rbr` 和就绪链表 `rdllist`）。
2. `epoll_ctl(EPOLL_CTL_ADD)`：将 socket fd 插入红黑树，同时向内核中断处理注册回调——当 fd 就绪时，回调函数将其加入 `rdllist`。
3. `epoll_wait()`：仅检查 `rdllist` 是否非空，有则返回。时间复杂度 O(1)。

#### 1.2.3 Redis 6.0 的多线程 I/O 模型

> 注意区分：**I/O 线程**（6.0+ 默认关闭） vs **命令执行线程**（始终单线程）

```
Redis 6.0+ 多线程 I/O 模型：

主线程：                    I/O 线程组：
┌──────────────┐           ┌────────┐
│ epoll_wait   │           │ IO-1   │ read(fd1, fd2)
│ 收集就绪事件  │──────┐    ├────────┤
│              │      │    │ IO-2   │ read(fd3,fd4)
│ 分发读任务   │      └──▶ │        │
│              │           └──┬─────┘
│ 单线程执行   │              │ 读完数据
│ 所有命令     │◀─────────────┘
│              │
│ 分发写任务   │──────┐    ┌────────┐
│              │      ├──▶ │ IO-1   │ write()
│              │      │    ├────────┤
│              │      └──▶ │ IO-2   │ write()
└──────────────┘           └────────┘

关键：命令执行（解析→查找→操作→响应构建）始终在主线程串行完成
```

**为什么默认不开启多线程 I/O？**
- 单机 QPS 不高的场景（< 5w/s），单线程处理网络 I/O 完全够用。
- 多线程 I/O 主要在**高并发 + 大 value 读写**时有明显收益。
- 配置：`io-threads 4` + `io-threads-do-reads yes`

### 1.3 优化的内部数据结构

Redis 没有直接使用 C++ STL 或通用库的数据结构，而是**自研了一套极度精简的实现**。

#### 1.3.1 SDS（Simple Dynamic String）—— 替代 C 原生字符串

**C 原生 `char*` 的问题：**
- 获取长度需要 O(n) 遍历 `\0`
- 拼接字符串容易缓冲区溢出
- 频繁修改时反复 `malloc/free`

**SDS 的设计（sds.h）：**

```
struct sdshdr {
    int len;    // 已用长度 —— O(1) 获取
    int free;   // 剩余可用空间
    char buf[]; // 柔性数组，实际数据
};
```

| 特性 | C 字符串 | SDS |
|------|---------|-----|
| 获取长度 | O(n) 遍历 | O(1) 读取 len |
| 缓冲区溢出 | 容易 | 自动扩容，杜绝溢出 |
| 内存分配 | 每次修改都要分配 | 空间预分配 + 惰性释放 |
| 二进制安全 | 遇到 `\0` 截断 | 用 len 记录长度，存任意二进制 |
| 兼容性 | — | 仍然以 `\0` 结尾，兼容 C 函数 |

**空间预分配（减少 malloc 次数）：**
- 修改后 len < 1MB → 分配 `len + free`，其中 `free = len`（总容量 = 2*len）
- 修改后 len ≥ 1MB → 额外分配 1MB

**惰性空间释放：**
- 缩短字符串时**不回收**内存，预留 free 空间，下次增长时直接复用。

#### 1.3.2 ziplist（压缩列表）—— 小数据的紧凑存储

ziplist 是 Redis 中最重要的紧凑数据结构，用于 **Hash / List / ZSet 在小数据量时** 的底层存储。

```
ziplist 内存布局（连续内存块，无指针）：

┌────────┬────────┬────────┬────────┬───────┬────────┐
│zlbytes │zltail  │zllen   │entry1  │entry2 │zlend   │
│4字节   │4字节   │2字节   │变长    │变长   │1字节   │
└────────┴────────┴────────┴────────┴───────┴────────┘
 总字节数  尾节点偏移 节点数                   结束标记 0xFF
```

**每个 entry 的结构：**

```
┌─────────────┬──────────────┬─────────┐
│ prevlen     │ encoding     │ data    │
│ 前驱节点长度 │ 类型+长度标记 │ 实际数据 │
└─────────────┴──────────────┴─────────┘
```

**为什么省内存？**
1. **连续内存，无指针开销。** 指针在 64 位系统上每个 8 字节，ziplist 用 prevlen 字段（1-5字节）替代，省掉了所有指针。
2. **变长编码。** encoding 字段按需使用，存小整数用 1 字节，存短字符串用 2 字节等。
3. **CPU 缓存友好。** 连续内存意味着更高的 cache line 命中率。

**但 ziplist 有明显的缺点——连锁更新：**
- 每个 entry 存了前驱长度 `prevlen`。如果前驱从 253 字节变成 254 字节，`prevlen` 需要从 1 字节扩展成 5 字节。
- 这可能导致当前 entry 也变长，进而触发下一个 entry 的 `prevlen` 扩展……一直传到末尾。
- 连锁更新的最坏时间复杂度是 O(N²)。**但实际很少触发**——因为触发条件苛刻（恰好卡在 253/254 临界值）。
- **Redis 7.0 用 listpack 取代了 ziplist，彻底解决了连锁更新问题。**

#### 1.3.3 skiplist（跳表）—— ZSet 的核心

ZSet 在数据量大时的存储结构是 **dict + skiplist**：

```
ZSet 结构：
┌─────────────────────────────────────┐
│  dict:  member → score              │  O(1) 查分，O(1) 去重
│  skiplist: score → member (有序链)  │  O(logN) 范围查询 / 排名
└─────────────────────────────────────┘
```

**为什么不用红黑树 / AVL 树？**

| 特性 | 跳表 | 红黑树 | AVL 树 |
|------|------|--------|--------|
| 范围查询 | O(logN)，只需遍历 | O(logN + M)，需要中序遍历非叶节点 | 同红黑树 |
| 实现复杂度 | 简单（200 行） | 复杂 | 中等 |
| 平衡维护 | 随机化，无需旋转 | 需要旋转/染色 | 需要旋转 |
| ZRANK 操作 | O(logN)，在节点中存 span | O(logN)，但需要额外维护 size | 同理 |
| 并发友好 | 可做 lock-free | 困难 | 困难 |

**Redis 跳表的核心结构：**

```
level 3:  HEAD ──────────────────────────► NULL
level 2:  HEAD ──────► 20 ─────────────► NULL
level 1:  HEAD ──► 10 ──► 20 ──► 30 ──► NULL
level 0:  HEAD ► 5 ► 10 ► 20 ► 30 ► 35 ► NULL
                   │         │
                   ▼         ▼
                "alice"   "bob"
               score:10  score:20

每个节点有一个 backward 指针（回退到前驱），方便逆序范围查询
每个 level 有一个 span 字段，记录该层到下一节点的跨度，用于 O(logN) 计算 rank
```

**跳表节点（zskiplistNode）核心字段：**
- `score`：分值（double）
- `ele`：成员对象（SDS）
- `backward`：后退指针（用于 ZREVRANGE）
- `level[]`：层级数组，每层包含 forward 指针和 span

**层数随机化：** 每个节点插入时，以 1/4 的概率递增层数（最多 64 层，即 ZSKIPLIST_MAXLEVEL）。这种幂次定律保证了期望复杂度 O(logN)。

#### 1.3.4 dict（渐进式 rehash）

Redis 的 hash 表（用于全局键空间、Hash 类型底层等）使用 **链地址法 + 渐进式 rehash**。

```
dict 结构：
┌──────────────────────────────────┐
│  ht[0]  ← 当前使用的哈希表       │
│  ht[1]  ← rehash 目标（平时为空）│
│  rehashidx ← -1 表示不在 rehash  │
└──────────────────────────────────┘
```

**渐进式 rehash 的步骤：**
1. 为 `ht[1]` 分配 `ht[0].used * 2` 的空间
2. 设置 `rehashidx = 0`
3. **每次对 dict 的增删改查操作，顺带迁移 `ht[0]` 的一个桶到 `ht[1]`**
4. 全部迁移完后，`ht[1]` 变成 `ht[0]`，释放旧表

**为什么不用一次性 rehash？**
- Redis 是单线程执行命令的。
- 如果键空间有几百万个 key，一次性 rehash 会导致 **秒级阻塞**。
- 渐进式 rehash 将耗时**均摊**到每个请求中，单次操作延迟几乎无感。

**rehash 期间的查找流程：**
```
查找 key：
1. 先在 ht[0] 中查找
2. 如果 rehashidx != -1（正在 rehash），再去 ht[1] 中查找
3. 新增操作只写 ht[1]
```

#### 1.3.5 整数集合（intset）

Set 类型在小数据且全为整数时，使用 intset 存储：

```
intset 结构：
┌──────────┬──────────┬─────────────────┐
│ encoding │ length   │ contents[]      │
│ 编码方式  │ 元素数量  │ 有序整数数组      │
└──────────┴──────────┴─────────────────┘
```

- 动态升级：全是 int16 时用 2 字节，插入一个 int32 时整个升级。
- **只升级不降级**，节省内存。

#### 1.3.6 quicklist（Redis 3.2+ 的 List 实现）

quicklist 是 **双向链表 + ziplist** 的混合结构：

```
quicklist：
┌─────────┐    ┌─────────┐    ┌─────────┐
│ quicklist│───▶│ quicklist│───▶│ quicklist│
│  node   │◀───│  node   │◀───│  node   │
│         │    │         │    │         │
│ ziplist │    │ ziplist │    │ ziplist │
│ [a,b,c] │    │ [d,e,f] │    │ [g,h,i] │
└─────────┘    └─────────┘    └─────────┘

每个节点存一个 ziplist，链表保证头尾操作 O(1)
ziplist 保证内存紧凑，减少碎片
通过 list-max-ziplist-size 控制每个 ziplist 的大小
```

#### 1.3.7 listpack（Redis 7.0+，取代 ziplist）

listpack 解决了 ziplist 的**连锁更新**问题：

```
ziplist:  entry 存 prevlen → 前置变长 → 连锁反应
listpack: entry 存 自身长度 → 无法从前驱推断 → 无连锁

listpack entry 结构：
┌──────────┬──────────┬──────────┬──────────┐
│ encoding │ data     │ backlen  │          │
│ 数据类型  │ 实际数据  │ 自身总长度 │          │
└──────────┴──────────┴──────────┴──────────┘

backlen 记录的是当前 entry 的总长度（用于反向遍历）
即使当前 entry 变长，也不影响下一个 entry 的任何字段
```

### 1.4 简洁高效的通信协议（RESP）

RESP = **REdis Serialization Protocol**

#### 1.4.1 协议设计原则

```
RESP 的 5 种数据类型：

1. Simple String   +OK\r\n
2. Error           -ERR unknown command\r\n
3. Integer         :1000\r\n
4. Bulk String     $6\r\nfoobar\r\n
5. Array           *2\r\n$3\r\nGET\r\n$3\r\nkey\r\n
```

**为什么 RESP 快？**

| 特性 | 说明 |
|------|------|
| **二进制安全** | Bulk String 通过 `$长度\r\n` 前缀指明长度，不依赖 `\0` 终止符 |
| **无 schema 开销** | 不需要 Protobuf/Thrift 的 IDL 编译，直接拼字符串 |
| **实现极简** | 解析器核心代码不到 200 行（`read` 前缀字节 → 分发到对应解析函数） |
| **人类可读** | 调试时可以直接 `nc` / `telnet` 敲命令 |
| **管道支持** | 协议本身就是自定界的，天然支持 pipeline（一次性发多条请求） |

#### 1.4.2 RESP 解析流程

```
客户端发送：*2\r\n$3\r\nGET\r\n$3\r\nkey\r\n

解析过程（状态机）：
1. 读到 '*'  → 进入"解析数组长度"状态 → 得到 2（两个元素）
2. 读到 '$'  → 进入"解析Bulk String长度"状态 → 得到 3
3. 读取 3 字节 + \r\n → "GET"
4. 读到 '$'  → 得到 3
5. 读取 3 字节 + \r\n → "key"
6. 数组长度匹配完成 → 执行命令

每次解析都是 O(N)，N = 请求字节数
```

#### 1.4.3 Pipeline（管道）

由于 RESP 天然自定界，客户端可以连续发送多个请求而不用等前一个的响应：

```
普通模式（一问一答）：
Client: GET a  ──────▶  Server
Client: ◀──────  "1"   Server
Client: GET b  ──────▶  Server
Client: ◀──────  "2"   Server

Pipeline 模式（批量发送）：
Client: GET a\nGET b\n ──▶  Server
Client: ◀────── "1"\n"2"   Server

RTT 节省：N 次请求从 N*RTT 变成 1*RTT
```

**面试要点：Pipeline 不是原子操作**——命令是排队执行的，中途其他 client 的命令可能插入。

## ⭐️ 2. 什么是 Redis Module？有什么用？

### 2.1 核心概念

**Redis Module 是 Redis 4.0 引入的插件化扩展机制**，允许第三方开发者通过 C 语言编写动态库（`.so` 文件），在 Redis 内核层面扩展新的数据类型、命令和功能，而无需修改 Redis 源码。

```
传统方式：改 Redis 源码 → 重新编译 → 替换二进制 → 重启
Module 方式：写 .so → redis.conf 配置 loadmodule → 动态加载
```

运行机制：Module 通过 Redis 提供的 **Modules API**（`redismodule.h`）与内核交互，运行在 Redis 主进程内，拥有与内置命令相同的执行权限和性能。

### 2.2 加载方式

```bash
# 方式一：配置文件
# redis.conf
loadmodule /path/to/module.so
loadmodule /path/to/another.so [arg...]

# 方式二：运行时加载
MODULE LOAD /path/to/module.so

# 查看已加载模块
MODULE LIST

# 卸载模块
MODULE UNLOAD modulename
```

### 2.3 核心价值

**1. 更快——运行在数据所在的进程内：**

```
网络往返对比：

不用 Module（传统架构）：          用 Module（就地计算）：
App ──▶ Redis GET data ──▶       App ──▶ Redis Module
App ◀── Redis return data        （内部计算、过滤、聚合）
App ──▶ 本地计算                      │
                                    ◀── 返回最终结果
需要多次网络 I/O                一次网络往返完成
```

**2. 更简单——原子性与分布式无关：**

一个 Module 命令在 Redis 单线程中一次性执行完毕，天然具备原子性，不需要分布式锁。

**3. 填补能力缺口：**

Redis 内置 5 种数据结构不是万能的，Module 可以创建**全新的、刚好适配业务的数据结构**。

### 2.4 知名 Redis Module 一览

| Module | 创造者 | 解决的问题 | 关键能力 |
|--------|-------|----------|---------|
| **RediSearch** | Redis Ltd | 全文搜索 + 向量检索 | 倒排索引引擎，支持模糊搜索、聚合过滤、向量相似度（KNN）。性能达到 Elasticsearch 的 4-6 倍 |
| **RedisJSON** | Redis Ltd | JSON 文档存储 | 允许在 JSON 文档内部进行原子级读写（`JSON.SET doc $.a.b[0] value`），支持 JSONPath 表达式 |
| **RedisTimeSeries** | Redis Ltd | 时序数据 | 专门为时间序列优化的双块存储，内置降采样、聚合、保留策略，比用 ZSet 模拟节省 60%+ 内存 |
| **RedisBloom** | Redis Ltd | 概率数据结构 | 布隆过滤器（空间效率 ~1%）、布谷鸟过滤器（支持删除）、Top-K（近似频率统计）、Count-Min Sketch |

### 2.5 RediSearch 实战

**全文搜索：**

```
FT.CREATE idx:articles ON HASH PREFIX 1 article: SCHEMA title TEXT WEIGHT 5.0 body TEXT
FT.SEARCH idx:articles "@title:redis @body:cache" LIMIT 0 10
```

**向量检索（AI 场景）：**

```
FT.CREATE idx:docs ON HASH PREFIX 1 doc: 
  SCHEMA text TEXT embedding VECTOR FLAT 6 DIM 768 DISTANCE_METRIC COSINE

FT.SEARCH idx:docs "*=>[KNN 10 @embedding $query_vec AS score]" 
  PARAMS 2 query_vec <768维二进制blob> SORTBY score RETURN 3 score text
```

### 2.6 面试题：Redis Module vs Lua 脚本？

| 维度 | Module | Lua 脚本 |
|------|--------|---------|
| **实现语言** | C 语言（或 Rust/Zig FFI） | Lua |
| **能力范围** | 完全自由——新数据结构、新命令、网络 I/O、AOF/RDB 自定义序列化 | 组合现有命令，不能创建新命令/新结构 |
| **性能上限** | 与内置 C 命令同级（纳秒级额外开销） | Lua → C 调用有上下文切换，复杂逻辑有 GC 暂停 |
| **持久化** | 自定义 AOF/RDB 序列化回调，数据可持久化 | 脚本本身不持久化数据，脚本内 SCRIPT LOAD 要单独管理 |
| **阻塞风险** | 长时间运行同样阻塞主线程，但可以内部 yield（`RedisModule_BlockClient`） | 执行期间完全阻塞，无法让出 CPU |
| **运维复杂度** | 编译 .so，服务器部署要慎重；用官方 Module（RediSearch 等）有完整文档 | `EVAL` / `EVALSHA` 一行命令即可，零运维成本 |

> **结论：** Lua 适合「把几个现有命令打包原子执行」（如分布式锁的加锁+设超时）。Module 适合「Redis 原生能力不够用」（如搜索、JSON 操作、时序存储、概率数据结构）。

## ⭐️ 3. 为什么用 Redis 而不用本地缓存？

这是大厂面试高频题，考察对**分布式系统 vs 单机系统**的理解深度。

### 3.1 本地缓存（Local Cache）是什么？

```
本地缓存：每个服务实例在进程内维护自己的缓存

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  Instance-1  │  │  Instance-2  │  │  Instance-3  │
│  ┌────────┐  │  │  ┌────────┐  │  │  ┌────────┐  │
│  │ Caffeine│  │  │  │ Caffeine│  │  │  │ Caffeine│  │
│  │ / Guava │  │  │  │ / Guava │  │  │  │ / Guava │  │
│  └────────┘  │  │  └────────┘  │  │  └────────┘  │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       │                 │                 │
       └─────────────────┴─────────────────┘
                         │
                    ┌────▼────┐
                    │   DB    │
                    └─────────┘
```

**优点：** 零网络延迟（纳秒级），无带宽瓶颈。
**典型技术：** Java 的 Caffeine/Guava、Go 的 bigcache/freecache、Python 的 lru_cache/functools。

### 3.2 对维度对比：Redis vs 本地缓存

| 维度 | Redis（分布式缓存） | 本地缓存（Caffeine/Guava） |
|------|-------------------|--------------------------|
| **数据一致性** | ✅ 单数据源，所有节点共享同一份缓存，写后删缓存，所有节点下一瞬间看到新版本 | ❌ 每个节点各存一份拷贝，更新后通知各节点刷新是分布式一致性问题（需要 MQ 广播 / 配置中心） |
| **内存利用率** | ✅ 全局共享，1GB 存 1GB 数据 | ❌ 5 节点 × 1GB = 5GB 堆内存，实际可能只存了 1GB 不重复数据 |
| **弹性 & 扩缩容** | ✅ 加节点不丢失缓存（缓存与业务分离） | ❌ 加节点冷启动，缓存从零开始；缩节点缓存消失 |
| **服务重启** | ✅ 缓存独立存活，服务重启缓存仍在，无冷启动 | ❌ 服务重启缓存全丢，DB 瞬间承受全量请求（缓存雪崩） |
| **隔离性** | ✅ 缓存层可独立扩容，不受业务进程内存限制 | ❌ 缓存与业务争用堆内存，GC 压力大；大缓存可能导致 Full GC |
| **数据多样性** | ✅ 5 种丰富数据结构 + Module | ❌ 基本只有 K-V（Map） |
| **持久化** | ✅ RDB/AOF，甚至当数据库用 | ❌ 进程重启全丢（除非手动序列化） |
| **访问延迟** | ⚠️ ~0.1-1ms（网络 RTT） | ✅ ~1-10ns（内存直接访问） |
| **运维复杂度** | ⚠️ 需要额外部署集群，需要监控网络 | ✅ 零运维，随业务进程启停 |

### 3.3 主流实践：多级缓存架构

```
┌──────────────────────────────────────────┐
│          多级缓存（分级策略）              │
│                                           │
│  ┌─────────┐                              │
│  │ L1 本地  │ Caffeine, 极热数据           │
│  │         │ 容量: ~1000 条, TTL: 1-5 秒  │
│  └────┬────┘                              │
│       │ miss                               │
│       ▼                                    │
│  ┌─────────┐                              │
│  │ L2 分布式│ Redis, 热点数据              │
│  │         │ 容量: ~100GB, TTL: 分钟~小时  │
│  └────┬────┘                              │
│       │ miss                               │
│       ▼                                    │
│  ┌─────────┐                              │
│  │ L3 持久化│ MySQL / PostgreSQL           │
│  │         │ 全量数据                      │
│  └─────────┘                              │
└──────────────────────────────────────────┘
```

**两级各有分工：**
- **L1 本地缓存：** 抗流量尖刺，缓存极热且变化少的 key（如配置、热点文章标题），TTL 要短（1-5 秒）来尽量减少一致性问题。
- **L2 Redis：** 真正扛大部分缓存的分布式缓存，容量大、持久化、数据结构丰富、所有节点共享。

### 3.4 选型决策树

```
是否需要多个节点共享同一份缓存数据？
    │
    ├── 否 → 需要丰富数据结构 / 持久化吗？
    │           ├── 否 → 本地缓存（Caffeine）
    │           └── 是 → Redis（单机也可用）
    │
    └── 是 → Redis（或 Redis + 本地多级）
                是否对网络延迟极度敏感？
                    ├── 否 → 纯 Redis
                    └── 是 → L1 本地 + L2 Redis 多级缓存
```

### 3.5 面试题：为什么不直接用 HashMap 而用 Redis？

**回答框架（分层呈现）：**

> **1. 分布式共享：** 现在互联网系统基本是分布式部署，一个服务多台实例。HashMap 只在当前 JVM 进程内可见，各实例的数据互相看不见——修改用户状态后，请求路由到另一台机器，读到的是过时数据。Redis 独立部署，所有服务实例共享同一个缓存视图，天然保证缓存一致性。
>
> **2. 独立存活、不丢数据：** 服务重启或发布时，JVM 进程销毁，HashMap 的数据全丢。重启后瞬间所有读请求穿透到数据库——冷启动缓存雪崩。Redis 独立部署，服务重启缓存还在（而且 Redis 本身有 RDB/AOF 持久化机制兜底）。
>
> **3. 内存弹性：** Java 堆内存除了存缓存，还要运行对象、处理 GC。大 HashMap 会导致堆膨胀，触发 Full GC 时整个服务暂停。Redis 独立部署，内存专门用于缓存，可以弹性扩缩——加 Redis 节点不影响业务进程。
>
> **4. 丰富能力：** HashMap 只能简单的 get/put，没有 TTL、没有原子操作、没有 Pub/Sub、没有 5 种数据结构、没有淘汰策略。这些 Redis 都有。
>
> **5. 但延迟敏感场景，本地缓存仍然是皇冠：** 本地缓存的访问延迟在纳秒级（直接内存读取），Redis 有 0.1-1ms 的网络 RTT。所以最成熟的方案是 **多级缓存**：L1 本地（Caffeine，TTL 很短）+ L2 Redis。

### ⭐️ 面试题：Redis 为什么这么快？

**标准回答框架（总分结构）：**

> Redis 之所以快，可以归纳为四个层面的设计：
>
> **第一，存储层面**——纯内存操作。所有数据都在内存中，读写延迟在纳秒到微秒级，比磁盘 I/O 快 3-4 个数量级。
>
> **第二，I/O 层面**——单线程事件循环 + I/O 多路复用。使用 epoll 监听大量连接，避免了多线程的锁竞争和上下文切换开销。值得一提的是，Redis 6.0 后引入了多线程 I/O，但命令执行仍保持单线程。
>
> **第三，数据结构层面**——自研的高效数据结构。比如 SDS 减少内存分配、ziplist/listpack 紧凑存储、跳表支持有序操作、渐进式 rehash 避免阻塞等，每个结构都是为特定场景深度定制的。
>
> **第四，协议层面**——RESP 协议极简，解析快，支持管道批量请求，减少了网络 RTT。

**面试官可能的追问：**

| 追问 | 你的应对 |
|------|---------|
| "单线程有什么缺点？" | 慢命令（KEYS、大集合的全量操作）会阻塞整个实例；无法利用多核 CPU。解决方案：SCAN 替代 KEYS、UNLINK 异步删除、分片集群。 |
| "Redis 6.0 多线程的动机？" | 瓶颈已从 CPU 转向网络 I/O（万兆网卡普及）。多线程只处理网络读写，命令执行仍是单线程，保持原子性。 |
| "epoll 为什么比 select 好？" | select 需要每次传入整个 fd_set 并全量遍历，O(n) 复杂度；epoll 通过红黑树 + 就绪链表实现事件驱动，只返回就绪 fd，O(1)。 |
| "跳表 vs 红黑树？" | 跳表实现简单（~200行）、范围查询更直观（只需顺链表遍历）、支持 O(logN) 的 rank 计算（通过 span 字段）。平衡树需要旋转维护，实现复杂得多。 |
| "什么是渐进式 rehash？" | Redis hash 表扩容时，将数据迁移分散到每次增删改查操作中，每个请求顺带迁移一个桶，避免一次性 rehash 导致的长时间阻塞。 |
